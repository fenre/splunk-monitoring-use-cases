# Authoritative ingest pipeline

This package downloads and normalises the external reference data the
regulatory-compliance catalogue depends on. The design invariants are:

1. **HTTPS-only fetching** with TLS 1.2+ and a stable User-Agent
   (`scripts/ingest/manifest.py`).
2. **Cryptographic provenance for every artefact** — every download is
   hashed (SHA-256) and recorded in `data/provenance/ingest-manifest.json`
   together with URL, byte count, HTTP status, ETag, and timestamp.
3. **Idempotence** — the shared `manifest.fetch()` helper caches vendor
   files under `vendor/*/` and will skip network I/O on repeated runs.
4. **Deterministic outputs** — normalised JSON is written with
   `sort_keys=True, indent=2`, so repeated runs are byte-identical unless
   upstream data changed.

## Drivers

| Driver              | Source                                           | Normalised output directory                    |
|---------------------|--------------------------------------------------|------------------------------------------------|
| `ingest_oscal.py`   | NIST OSCAL catalogues (CSF 2.0, SP 800-53 rev5 + baselines, SP 800-171 rev3, SP 800-218 SSDF) | `data/crosswalks/oscal/`       |
| `ingest_attack.py`  | MITRE ATT&CK Enterprise, ICS, Mobile STIX bundles | `data/crosswalks/attack/`                      |
| `ingest_d3fend.py`  | MITRE D3FEND ontology + full mappings             | `data/crosswalks/d3fend/`                      |
| `ingest_atomic.py`  | Red Canary Atomic Red Team master index           | `data/crosswalks/atomic-red-team/`             |
| `ingest_olir.py`    | Cross-framework OLIR-style mappings via CTID Mappings Explorer (NIST 800-53, CRI Profile, CSA CCM) | `data/crosswalks/olir/`                        |

## Running the full pipeline

```bash
.venv-feasibility/bin/python scripts/ingest_all.py
```

This orchestrator runs each driver in order; any failure exits non-zero.

## Vendor & output directories

```
vendor/
├── attack/           # STIX bundles
├── atomic-red-team/  # atomics master index
├── d3fend/           # JSON-LD ontology + full mappings
├── olir/             # CTID Mappings Explorer JSON files
└── oscal/            # NIST OSCAL catalogues / profiles

data/
└── crosswalks/       # normalised JSON used by downstream consumers
    ├── attack/
    ├── atomic-red-team/
    ├── d3fend/
    ├── olir/
    └── oscal/
```

## Manifest

```
data/provenance/
└── ingest-manifest.json   # SHA-256 provenance ledger for every fetched file
```

Do not edit this file by hand — it is regenerated on every ingest run.

## Why CTID Mappings Explorer for OLIR

Authoritative NIST OLIR submissions (https://csrc.nist.gov/projects/olir)
are published primarily as downloadable XLSX spreadsheets behind the
OLIR web UI, which makes them unsuitable for a reproducible,
hash-verified ingest pipeline. The Center for Threat-Informed Defense
(CTID) re-publishes equivalent cross-framework crosswalks in stable,
machine-readable JSON at
<https://github.com/center-for-threat-informed-defense/mappings-explorer>
under a Creative Commons licence (see `LEGAL.md`). That repository is
the highest-quality open-source analogue to NIST OLIR and is therefore
used as our primary source. Future revisions can add more crosswalks by
appending entries to `SOURCES` in `ingest_olir.py`.
