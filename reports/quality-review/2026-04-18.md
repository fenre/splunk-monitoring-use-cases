# Catalog Quality & Consistency Review — 2026-04-18

**Method:** 26 parallel reviewer agents, one per category (cat-10 and cat-22 sub-sharded into 3 each). Each agent applied a 14-point semantic rubric to flag issues that existing automated audits (`audit_uc_structure.py`, `audit_spl_hallucinations.py`, `audit_quality_metadata.py`, etc.) cannot detect.

**Scope:** 6,424 UCs across 23 category files (~172k lines of markdown).

**Focus:** SPL correctness, CIM-SPL alignment, Value/Implementation vagueness, thresholds, near-duplicates, nomenclature drift, and known-FP gaps. Mechanical checks (missing fields, invalid IDs, unknown sourcetypes) are explicitly out of scope.

---

## Executive summary

### Top cross-cutting patterns (reported by ≥6 shards)

1. **CIM SPL is systematically decoupled from primary SPL.** In nearly every category with optional CIM SPL blocks, reviewers flagged the companion `tstats` as a *generic template* (Performance memory/swap, `Authentication.action=failure`, `Network_Traffic` bytes, `Endpoint.Processes`) that does not mirror the fields or semantics of the primary search. Worst offenders: cat-01, cat-02, cat-06, cat-07, cat-10 (§10.13), cat-14 (§14.9), cat-17, cat-18, cat-19. This is the single largest quality defect in the catalog.

2. **`stats … by _time span=X` is invalid SPL and appears across 10+ categories.** Found in cat-02, cat-06, cat-07, cat-08 (x2), cat-13, cat-15, cat-17 (x2), cat-18 (x2), cat-20. `span=` only belongs on `bin` or `timechart`. This should be mechanically detectable and added to `audit_spl_hallucinations.py` or a new SPL-grammar linter.

3. **Known false positives is effectively blank across most of the catalog.** cat-01 has it on exactly **one** UC (UC-1.1.1); cat-07, cat-12, cat-16, cat-20, cat-23 have it on **zero**. Security-adjacent categories (cat-09 IAM, cat-10) have either blank placeholders (`|`) or the same copy-pasted "administrative tasks / scheduled jobs / platform updates" paragraph pasted onto unrelated UCs. For the 5 most noise-prone categories (9, 10, 11, 14, 22) this is a HIGH-severity systemic gap.

4. **ESCU/Risk-based SPL decoupled from stated Data Sources.** cat-10 (§10.1–§10.9) and parts of §10.11 frequently show a body SPL of only `| from datamodel=Risk.All_Risk …` while **Data Sources** name raw vendor feeds (ASA, FTD, CloudTrail, Duo). Without the upstream ESCU correlation searches populating Risk, the shown SPL cannot evidence the detection. ~100s of UCs affected.

5. **Threshold numbers stated in Implementation prose not implemented in SPL.** Windows like "5 failures in 15 min", "80% capacity", "3σ anomaly" appear in Implementation text but the SPL uses unbounded aggregations without `bin`/`span` or uses arbitrary fixed thresholds. Found in cat-01, cat-04, cat-05, cat-08, cat-09, cat-10, cat-13, cat-14, cat-20.

6. **`Risk.All_Risk` boilerplate + `Table, Timeline (from ESCU)` visualization** dominates cat-10.6 and cat-10.7 — hundreds of near-identical SPL blocks that differ only in title/CVE/technique metadata. Consider flagging these as "ESCU-mirror" UCs with a pointer instead of near-duplicate SPL.

7. **Near-duplicate UCs across vendors.** IAM (Okta impossible travel = UC-9.3.2 ≈ UC-9.5.11; OAuth grants = UC-9.3.9 ≈ UC-9.5.10), EDR (identical malware detection across 5 vendors in §10.3), SIEM (Splunk ES vs Sentinel vs QRadar in §10.7), HCI (Nutanix Prism Central vs generic hci:* in cat-19), SWG block patterns (cat-17 §17.3), cold-chain excursion (cat-21 §21.3.5 ≈ §21.4.10).

8. **Index/sourcetype naming drift within a single file.** Examples: `index=cisco_aci` vs `index=aci` (cat-18), `index=k8s` vs `index=containers` (cat-03), `index=azure` + `mscs:azure:*` vs `index=cloud` + `azure:*` (cat-04), `index=hci` vs `index=nutanix` (cat-19), `index=finops` vs `index=cloud_cost` (cat-20). This breaks copy-paste reproducibility.

9. **Monitoring type mis-labelled as "Performance" for security/compliance UCs.** cat-09 has break-glass usage, suspicious sessions, geofencing, password policy violations tagged Performance (all 4 flagged HIGH). cat-15 door open/close monitoring also mistagged.

10. **Visualization section is chart-type boilerplate.** "Table, Bar chart, Line chart" repeats across hundreds of UCs without specifying axes, split-by fields, time grain, drilldown tokens, or panel-specific fields — flagged in 20/26 shards.

### Recommended actions (ordered by leverage)

1. **Build an SPL-grammar linter** that catches `stats ... span=`, `| mstats` without index scope, `case()` with wildcard string args, post-`timechart` filtering on fields that no longer exist, `join` without `subsearch` time bounds. One script would eliminate ~10% of findings.

2. **Decide CIM SPL policy** — either (a) drop optional CIM SPL sections entirely from UCs whose data isn't CIM-normalized and cannot be proven to map cleanly, or (b) enforce that CIM SPL must query the same object(s) declared in `CIM Models` and aggregate comparable signals. Either choice resolves ~150 findings.

3. **Known-FP guidance policy.** Promote Known false positives from optional to required for noisy detection categories (cat-9, 10, 11, 14, 17, 22). Forbid reuse of the "admin tasks / scheduled jobs" boilerplate across unrelated UCs.

4. **Consolidate ESCU-mirror UCs** in §10.6–§10.7. Replace the ~900 `Risk.All_Risk` wrappers with a single canonical "ESCU analytic surface" pattern + link table, or clearly mark them as metadata-only references.

5. **Freeze index/sourcetype naming conventions per vendor/product** and run a linter that flags deviations. Specifically: ACI, NSX, Nutanix, Kubernetes, Azure, finops indices.

6. **Scrub Monitoring type labels** — every UC currently tagged `Performance` that is about access, policy violation, or security events should be audited and re-tagged.

## Methodology

**Rubric (14 semantic checks, all agents shared the same):**
1. SPL-vs-DataSources mismatch
2. CIM-SPL vs CIM-Models mismatch
3. Tautological/vague Value
4. Vague Implementation (no actionable setup, missing thresholds)
5. Generic Visualization
6. Threshold issues (unjustified, impossible, or inconsistent with prose)
7. Criticality/difficulty mismatch (across similar UCs or internal SPL complexity)
8. Placeholder text (TBD, FIXME, TODO, example.com)
9. Known-FP gap on high-noise UCs
10. Near-duplicate SPL within the shard
11. Title/content mismatch
12. SPL anti-patterns (unfiltered `*`, subsearch-instead-of-tstats, join-instead-of-stats)
13. Nomenclature drift within the shard (field names, index names)
14. Stale references (deprecated products, removed features)
