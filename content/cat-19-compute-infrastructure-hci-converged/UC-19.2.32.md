# UC-19.2.32 · VxRail vCenter Extension and Marvin Plugin Health

## Description

VxRail management extensions integrate cluster operations with vCenter. Plugin or API failures block LCM workflows and mask host issues from operators who rely on the VxRail UI.

## Value

VxRail management extensions integrate cluster operations with vCenter. Plugin or API failures block LCM workflows and mask host issues from operators who rely on the VxRail UI.

## Implementation

(1) Collect plugin and internal API health from VxRail Manager or automation probes; (2) alert on unhealthy state, stale heartbeat, or rising API errors; (3) correlate with vCenter service restarts and SSO certificate rotations.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Custom (VxRail Manager REST API), vCenter logs.
• Ensure the following data sources are available: `index=vxrail` `sourcetype="vxrail:plugin"` with fields `cluster_id`, `plugin_state`, `last_heartbeat`, `api_error_count`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Collect plugin and internal API health from VxRail Manager or automation probes; (2) alert on unhealthy state, stale heartbeat, or rising API errors; (3) correlate with vCenter service restarts and SSO certificate rotations.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=vxrail sourcetype="vxrail:plugin" earliest=-24h
| eval hb_age_sec=now()-last_heartbeat
| where plugin_state!="healthy" OR hb_age_sec>600 OR api_error_count>0
| stats latest(plugin_state) as st, max(hb_age_sec) as max_hb_lag, sum(api_error_count) as errs by cluster_id
| sort -errs
```
