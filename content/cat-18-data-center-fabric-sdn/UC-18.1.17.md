# UC-18.1.17 · ACI Multisite Health

## Description

Monitors inter-site control-plane sync, spine proxy, and state for Cisco ACI Multi-Site / Multi-Pod deployments.

## Value

Monitors inter-site control-plane sync, spine proxy, and state for Cisco ACI Multi-Site / Multi-Pod deployments.

## Implementation

Ingest MSO/NDO or per-APIC multisite diagnostics. Alert on any site not `in-sync`. Runbook for partition scenarios.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: `TA_cisco-ACI`, MSO/APIC cross-site events.
• Ensure the following data sources are available: `cisco:aci:multisite`, APIC syslog.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Ingest MSO/NDO or per-APIC multisite diagnostics. Alert on any site not `in-sync`. Runbook for partition scenarios.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=cisco_aci sourcetype="cisco:aci:multisite" earliest=-24h
| where match(lower(status),"(?i)out.of.sync|isolated|failed|degraded")
| stats count by site_name, peer_site, component
| sort -count
```

Understanding this SPL

**ACI Multisite Health** — Monitors inter-site control-plane sync, spine proxy, and state for Cisco ACI Multi-Site / Multi-Pod deployments.

Documented **Data sources**: `cisco:aci:multisite`, APIC syslog. **App/TA** (typical add-on context): `TA_cisco-ACI`, MSO/APIC cross-site events. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: cisco_aci; **sourcetype**: cisco:aci:multisite. That sourcetype matches what this use case lists under Data sources.

**Pipeline walkthrough**

• Scopes the data: index=cisco_aci, sourcetype="cisco:aci:multisite", time bounds. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Filters the current rows with `where match(lower(status),"(?i)out.of.sync|isolated|failed|degraded")` — typically the threshold or rule expression for this monitoring goal.
• `stats` rolls up events into metrics; results are split **by site_name, peer_site, component** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
