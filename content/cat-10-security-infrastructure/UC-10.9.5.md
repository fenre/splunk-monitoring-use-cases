# UC-10.9.5 · Prompt extraction defense evasion

## Description

This detection aligns with ESCU 2025–2026 analytic story coverage for: Prompt extraction defense evasion. Detects attempts to extract system prompts, policies, or hidden instructions from chat interfaces.

## Value

## Implementation

ESCU Hunting detection — not designed for automated alerting. Run ad-hoc or on a low-frequency schedule from Splunk Enterprise Security for analyst-driven investigation. Requires LLM gateway logs, application security logs ingested and CIM-normalized.

## Detailed Implementation

Enterprise Security Detection Rule

"Prompt extraction defense evasion" is a Hunting detection designed for analyst-driven threat hunting rather than automated alerting, sourced from the Splunk Enterprise Security Content Update (ESCU). Hunting detections are not intended for automated alerting. They are run on-demand or on a low-frequency schedule by security analysts investigating specific threat hypotheses.

Prerequisites

• Splunk Enterprise Security 7.x or later with the ES Content Update (ESCU) app installed and up to date.
• Data sources: LLM gateway logs, application security logs. Must be ingested into Splunk and normalized to the Common Information Model (CIM) via the appropriate Technology Add-on.
• Security domain (threat): Configure threat intelligence feeds and verify the Threat Intelligence data model is populated for IOC matching.
• MITRE ATT&CK: T1552.004, T1562. Review the ATT&CK matrix for adjacent techniques to identify detection coverage gaps in your environment.

Deployment

1. In Enterprise Security, navigate to Configure → Content → Content Management.
2. Search for "Prompt extraction defense evasion" or filter by Analytic Story.
3. Hunting detections are typically left disabled for automated scheduling. Instead, run them on-demand from the Search bar or schedule at low frequency (daily or weekly).
4. Review results manually — hunting detections cast a wider net and expect analyst judgment to separate signal from noise.
5. When results warrant further investigation, create a Notable Event manually or initiate your incident response workflow.

Tuning and False Positive Management

Known false positives for this detection: Broad scope: expect noise. Triage using asset criticality, user role and recent change-management activity; promote to a detection only after stable tuning.

• Hunting detections are expected to produce broader result sets that require analyst interpretation. Focus on refining the search scope (time range, specific hosts or users) rather than eliminating all noise.
• Maintain a hunting journal documenting hypotheses tested, results found, and detection improvements identified.
• Share high-fidelity findings with the detection engineering team to convert recurring hunting patterns into automated TTP or Anomaly detections.

Analyst Response Workflow

When reviewing Hunting results:

1. Run the search manually or review scheduled results. Evaluate each returned event against your threat hypothesis — not every result indicates compromise.
2. Cross-reference findings with current threat intelligence, recent security advisories, and outputs from other detection sources.
3. If suspicious activity is confirmed, create a Notable Event or escalate directly to your incident response process with the supporting evidence.
4. Document the hunting exercise: hypothesis, data sources queried, time range, findings, and any detection engineering improvements identified.

About the SPL Query Shown Above

The SPL displayed for this use case is the Risk Investigation drilldown search — it queries the Risk data model to show all risk events associated with a specific entity. This is the search analysts use during investigation to understand what contributed to an entity’s risk score.
