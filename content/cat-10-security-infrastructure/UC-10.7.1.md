# UC-10.7.1 · Alert Volume Trending

## Description

Alert volume trends reveal SOC workload, detection rule effectiveness, and potential alert fatigue risks.

## Value

Alert volume trends reveal SOC workload, detection rule effectiveness, and potential alert fatigue risks.

## Implementation

Track notable event volume from ES over time. Break down by source (correlation search). Identify noisy rules for tuning. Alert when daily volume exceeds analyst capacity thresholds. Report on volume trends.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security.
• Ensure the following data sources are available: ES notable events (`` `notable` `` macro).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Track notable event volume from ES over time. Break down by source (correlation search). Identify noisy rules for tuning. Alert when daily volume exceeds analyst capacity thresholds. Report on volume trends.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
`notable`
| timechart span=1d count by source
| sort -count
```

Understanding this SPL

**Alert Volume Trending** — Alert volume trends reveal SOC workload, detection rule effectiveness, and potential alert fatigue risks.

Documented **Data sources**: ES notable events (`` `notable` `` macro). **App/TA** (typical add-on context): Splunk Enterprise Security. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

**Pipeline walkthrough**

• Invokes macro `notable` — in Search, use the UI or expand to inspect the underlying SPL.
• `timechart` plots the metric over time using **span=1d** buckets with a separate series **by source** — ideal for trending and alerting on this use case.
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.


Step 3 — Validate
Confirm that events are present in the index and that the search returns expected results. Compare with known good/bad scenarios if applicable. Verify field extractions and index permissions.

Step 4 — Operationalize
Add the search to a dashboard or set up alert actions (email, webhook, PagerDuty, etc.) as required. Document the use case in your runbook and assign an owner. Consider visualizations: Stacked area (alerts by source), Line chart (total alert volume), Bar chart (top alerting rules).
