# UC-10.12.20 · Breach Notification Readiness Assessment

## Description

Measures ability to identify affected individuals within HIPAA breach notification timelines.

## Value

Measures ability to identify affected individuals within HIPAA breach notification timelines.

## Implementation

Run tabletop data queries; pre-build patient cohort lists from message IDs and chart exports.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: DLP, EHR, email gateway.
• Ensure the following data sources are available: `sourcetype="dlp:phi"`, `sourcetype="o365:mail"`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Run tabletop data queries; pre-build patient cohort lists from message IDs and chart exports.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=security (sourcetype="dlp:phi" OR (sourcetype="o365:mail" severity="high"))
| search match(body,"(?i)(ssn|mrn|diagnosis)")
| stats earliest(_time) as first_seen, latest(_time) as last_seen, dc(recipient) as recipients by incident_id
| eval dwell_hours=(last_seen-first_seen)/3600
| where dwell_hours>24
```

Understanding this SPL

**Breach Notification Readiness Assessment** — Measures ability to identify affected individuals within HIPAA breach notification timelines.

Documented **Data sources**: `sourcetype="dlp:phi"`, `sourcetype="o365:mail"`. **App/TA** (typical add-on context): DLP, EHR, email gateway. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: security; **sourcetype**: dlp:phi, o365:mail. Those sourcetypes align with what this use case lists under Data sources.

**Pipeline walkthrough**

• Scopes the data: index=security, sourcetype="dlp:phi". Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Applies an explicit `search` filter to narrow the current result set.
• `stats` rolls up events into metrics; results are split **by incident_id** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• `eval` defines or adjusts **dwell_hours** — often to normalize units, derive a ratio, or prepare for thresholds.
• Filters the current rows with `where dwell_hours>24` — typically the threshold or rule expression for this monitoring goal.
