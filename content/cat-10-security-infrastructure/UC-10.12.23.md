# UC-10.12.23 · Medical Device Network Segmentation

## Description

Detects east-west traffic from device VLANs to corporate or internet against segmentation policy.

## Value

Detects east-west traffic from device VLANs to corporate or internet against segmentation policy.

## Implementation

Document clinical exceptions (vendor VPN); use dynamic lists for imaging modalities.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Firewall, NAC.
• Ensure the following data sources are available: `sourcetype="pan:traffic"`, `sourcetype="ise:radius"`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Document clinical exceptions (vendor VPN); use dynamic lists for imaging modalities.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=network sourcetype="pan:traffic" src_zone="MED_DEVICE"
| lookup allowed_med_flows src dest_zone OUTPUT approved
| where approved!="true" AND dest_zone!="PACS"
| stats count by src, dest, dest_port, app
| sort -count
```

Understanding this SPL

**Medical Device Network Segmentation** — Detects east-west traffic from device VLANs to corporate or internet against segmentation policy.

Documented **Data sources**: `sourcetype="pan:traffic"`, `sourcetype="ise:radius"`. **App/TA** (typical add-on context): Firewall, NAC. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

**Pipeline walkthrough**

• Scopes the data: index=network, sourcetype="pan:traffic". Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Enriches events using `lookup` (lookup definition + optional OUTPUT fields).
• Filters the current rows with `where approved!="true" AND dest_zone!="PACS"` — typically the threshold or rule expression for this monitoring goal.
• `stats` rolls up events into metrics; results are split **by src, dest, dest_port, app** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
