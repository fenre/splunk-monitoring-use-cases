# UC-10.8.39 · Security Tool Availability and Heartbeat

## Description

Security tools that stop reporting indicate outage or tampering. Heartbeat monitoring ensures continuous visibility and alerts on missing data.

## Value

Security tools that stop reporting indicate outage or tampering. Heartbeat monitoring ensures continuous visibility and alerts on missing data.

## Implementation

Use Splunk heartbeat or last-seen per host/sourcetype. Alert when any critical source has not reported for >15 minutes. Maintain source inventory and expected reporting interval.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: All security data source TAs.
• Ensure the following data sources are available: Heartbeat or last-event timestamp per source type.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Use Splunk heartbeat or last-seen per host/sourcetype. Alert when any critical source has not reported for >15 minutes. Maintain source inventory and expected reporting interval.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=_internal source=*heartbeat* OR index=main sourcetype=*
| stats latest(_time) as last_seen by host, sourcetype
| eval gap=now()-last_seen
| where gap > 900
| table host, sourcetype, last_seen, gap
```

Understanding this SPL

**Security Tool Availability and Heartbeat** — Security tools that stop reporting indicate outage or tampering. Heartbeat monitoring ensures continuous visibility and alerts on missing data.

Documented **Data sources**: Heartbeat or last-event timestamp per source type. **App/TA** (typical add-on context): All security data source TAs. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: _internal, main; **sourcetype**: *. That sourcetype matches what this use case lists under Data sources.

**Pipeline walkthrough**

• Scopes the data: index=_internal, index=main, sourcetype=*. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `stats` rolls up events into metrics; results are split **by host, sourcetype** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• `eval` defines or adjusts **gap** — often to normalize units, derive a ratio, or prepare for thresholds.
• Filters the current rows with `where gap > 900` — typically the threshold or rule expression for this monitoring goal.
• Pipeline stage (see **Security Tool Availability and Heartbeat**): table host, sourcetype, last_seen, gap
