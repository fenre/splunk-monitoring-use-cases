# UC-10.6.142 · Citrix ADC Exploitation CVE-2023-3519

## Description

The following analytic identifies potential exploitation attempts against Citrix ADC related to CVE-2023-3519. It detects POST requests to specific web endpoints associated with this vulnerability by leveraging the Web datamodel. This activity is significant as CVE-2023-3519 involves a SAML processing overflow issue that can lead to memory corruption, posing a high risk. ## Detailed Implementation

Enterprise Security Detection Rule

"Citrix ADC Exploitation CVE-2023-3519" is a Hunting detection designed for analyst-driven threat hunting rather than automated alerting, sourced from the Splunk Enterprise Security Content Update (ESCU). Hunting detections are not intended for automated alerting. They are run on-demand or on a low-frequency schedule by security analysts investigating specific threat hypotheses.

Prerequisites

• Splunk Enterprise Security 7.x or later with the ES Content Update (ESCU) app installed and up to date.
• Data sources: Palo Alto Network Threat. Must be ingested into Splunk and normalized to the Common Information Model (CIM) via the appropriate Technology Add-on.
• MITRE ATT&CK: T1190. Review the ATT&CK matrix for adjacent techniques to identify detection coverage gaps in your environment.

Deployment

1. In Enterprise Security, navigate to Configure → Content → Content Management.
2. Search for "Citrix ADC Exploitation CVE-2023-3519" or filter by Analytic Story.
3. Hunting detections are typically left disabled for automated scheduling. Instead, run them on-demand from the Search bar or schedule at low frequency (daily or weekly).
4. Review results manually — hunting detections cast a wider net and expect analyst judgment to separate signal from noise.
5. When results warrant further investigation, create a Notable Event manually or initiate your incident response workflow.

Tuning and False Positive Management

Known false positives for this detection: False positives may be present based on organization use of SAML utilities. Filter, or restrict the analytic to Citrix devices only.

• Hunting detections are expected to produce broader result sets that require analyst interpretation. Focus on refining the search scope (time range, specific hosts or users) rather than eliminating all noise.
• Maintain a hunting journal documenting hypotheses tested, results found, and detection improvements identified.
• Share high-fidelity findings with the detection engineering team to convert recurring hunting patterns into automated TTP or Anomaly detections.

Analyst Response Workflow

When reviewing Hunting results:

1. Run the search manually or review scheduled results. Evaluate each returned event against your threat hypothesis — not every result indicates compromise.
2. Cross-reference findings with current threat intelligence, recent security advisories, and outputs from other detection sources.
3. If suspicious activity is confirmed, create a Notable Event or escalate directly to your incident response process with the supporting evidence.
4. Document the hunting exercise: hypothesis, data sources queried, time range, findings, and any detection engineering improvements identified.

## SPL

```spl
| tstats count min(_time) as firstTime max(_time) as lastTime FROM datamodel=Web
      WHERE Web.url IN ("*/saml/login","/cgi/samlauth","*/saml/activelogin","/cgi/samlart?samlart=*","*/cgi/logout","/gwtest/formssso?event=start&target=*","/netscaler/ns_gui/vpn/*")  Web.http_method=POST
      BY Web.http_user_agent, Web.status Web.http_method,
         Web.url, Web.url_length, Web.src,
         Web.dest, sourcetype
    | `drop_dm_object_name("Web")`
    | `security_content_ctime(firstTime)`
    | `security_content_ctime(lastTime)`
    | `citrix_adc_exploitation_cve_2023_3519_filter`
```
