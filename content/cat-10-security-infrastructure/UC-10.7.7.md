# UC-10.7.7 · AWS Defense Evasion Delete CloudTrail

## Description

Detects the deletion of AWS CloudTrail logs via DeleteTrail events. Adversaries may delete CloudTrail to evade detection and operate stealthily. Confirmed malicious activity could allow attackers to cover their tracks and prolong unauthorized access.

## Value

Detects the deletion of AWS CloudTrail logs via DeleteTrail events. Adversaries may delete CloudTrail to evade detection and operate stealthily. Confirmed malicious activity could allow attackers to cover their tracks and prolong unauthorized access.

## Implementation

Install Splunk AWS Add-on and enable CloudTrail logs in your AWS environment. Deploy the ESCU detection from Splunk Security Essentials or security_content.

## Detailed Implementation

Enterprise Security Detection Rule

"AWS Defense Evasion Delete CloudTrail" is a TTP (Tactics, Techniques, and Procedures) detection that identifies known adversary behaviors with high fidelity, sourced from the Splunk Enterprise Security Content Update (ESCU). When triggered, it creates a Notable Event in the Incident Review dashboard for analyst triage.

Prerequisites

• Splunk Enterprise Security 7.x or later with the ES Content Update (ESCU) app installed and up to date.
• Data sources: AWS CloudTrail DeleteTrail. Must be ingested into Splunk and normalized to the Common Information Model (CIM) via the appropriate Technology Add-on.
• Security domain (cloud): Ensure data relevant to the cloud security domain is ingested and CIM-normalized.
• MITRE ATT&CK: T1562.008. Review the ATT&CK matrix for adjacent techniques to identify detection coverage gaps in your environment.

Deployment

1. In Enterprise Security, navigate to Configure → Content → Content Management.
2. Search for "AWS Defense Evasion Delete CloudTrail" or filter by Analytic Story.
3. Review the detection configuration — verify the scheduling interval and throttling settings match your operational tempo.
4. Enable the detection as a Correlation Search. It will create Notable Events directly when triggered.
5. Set the Notable Event severity and urgency appropriate to your environment’s risk posture.
6. Configure Adaptive Response Actions: email notifications, ServiceNow ticket creation, SOAR playbook triggers, or other response workflows.

Tuning and False Positive Management

Known false positives for this detection: Legitimate teardown of CloudTrail by an admin; verify with change management.

• Review the detection’s filter criteria and adjust for known-good activity in your environment.
• Configure throttling in Content Management to prevent duplicate Notable Events for the same entity within a configurable window (typically 1–24 hours depending on detection frequency).
• Use Notable Event Suppression for entities or patterns that are consistently benign after investigation.

Analyst Response Workflow

When this detection generates a Notable Event:

1. Open the Notable Event in Incident Review. Review the triggering event details, affected entities, and assigned severity.
2. Investigate the involved entities using the Asset Investigator and Identity Investigator dashboards for historical context and behavioral patterns.
3. Correlate with related Notable Events and threat intelligence to assess whether this is an isolated event or part of a broader campaign.
4. Take appropriate response actions: contain, remediate, and recover. Leverage SOAR playbooks where available for consistent and rapid response.
5. Update the Notable Event status and document investigation findings for post-incident review and compliance.
