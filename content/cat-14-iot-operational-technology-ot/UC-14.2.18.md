# UC-14.2.18 · Engineering Workstation Anomaly

## Description

USB inserts, new binaries, or remote sessions on EWS workstations are high-risk in OT.

## Value

USB inserts, new binaries, or remote sessions on EWS workstations are high-risk in OT.

## Implementation

Lock down EWS to approved paths. Alert on new process or driver load. Correlate with maintenance windows.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: EDR on EWS, Windows Security / Sysmon.
• Ensure the following data sources are available: `sourcetype="sysmon:windows"` `tag=ews`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Lock down EWS to approved paths. Alert on new process or driver load. Correlate with maintenance windows.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=ot sourcetype="sysmon:windows" host_tag="EWS"
| search EventCode=1 OR EventCode=11
| where NOT match(Image,"(?i)(approved\\\\path)")
| stats count by Computer, Image, ParentImage
| sort -count
```

Understanding this SPL

**Engineering Workstation Anomaly** — USB inserts, new binaries, or remote sessions on EWS workstations are high-risk in OT.

Documented **Data sources**: `sourcetype="sysmon:windows"` `tag=ews`. **App/TA** (typical add-on context): EDR on EWS, Windows Security / Sysmon. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: ot; **sourcetype**: sysmon:windows. That sourcetype matches what this use case lists under Data sources.

**Pipeline walkthrough**

• Scopes the data: index=ot, sourcetype="sysmon:windows". Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Applies an explicit `search` filter to narrow the current result set.
• Filters the current rows with `where NOT match(Image,"(?i)(approved\\\\path)")` — typically the threshold or rule expression for this monitoring goal.
• `stats` rolls up events into metrics; results are split **by Computer, Image, ParentImage** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
