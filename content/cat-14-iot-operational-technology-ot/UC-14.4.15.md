# UC-14.4.15 · MQTT Broker Overload

## Description

Broker CPU, connection count, and retained message backlog indicate need to shard or scale.

## Value

Broker CPU, connection count, and retained message backlog indicate need to shard or scale.

## Implementation

Scrape Prometheus or vendor API. Alert on sustained high utilization or any dropped messages. Correlate with misbehaving clients publishing at high QoS0 rate.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Mosquitto/HiveMQ/AWS IoT metrics.
• Ensure the following data sources are available: `sourcetype="mqtt:broker_metrics"`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Scrape Prometheus or vendor API. Alert on sustained high utilization or any dropped messages. Correlate with misbehaving clients publishing at high QoS0 rate.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=iot sourcetype="mqtt:broker_metrics"
| where connections > max_connections*0.9 OR cpu_pct > 85 OR dropped_messages > 0
| timechart span=1m avg(connections) as conn, avg(cpu_pct) as cpu, sum(dropped_messages) as drops
```
