# UC-14.2.4 · Network Segmentation Monitoring

## Description

IT/OT network boundary violations create cybersecurity risk to critical infrastructure. Continuous monitoring validates segmentation.

## Value

IT/OT network boundary violations create cybersecurity risk to critical infrastructure. Continuous monitoring validates segmentation.

## Implementation

Forward IT/OT boundary firewall logs. Monitor all traffic crossing the boundary. Alert on unexpected protocols or connections. Validate against whitelist of approved communications. Report for ICS security audits.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Firewall TAs, network flow data.
• Ensure the following data sources are available: Industrial firewall logs, network flow data at IT/OT boundary.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Forward IT/OT boundary firewall logs. Monitor all traffic crossing the boundary. Alert on unexpected protocols or connections. Validate against whitelist of approved communications. Report for ICS security audits.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=network sourcetype="pan:traffic" zone_pair="IT-to-OT"
| where action="allow"
| stats count by src, dest, dest_port, app
| sort -count
```

Understanding this SPL

**Network Segmentation Monitoring** — IT/OT network boundary violations create cybersecurity risk to critical infrastructure. Continuous monitoring validates segmentation.

Documented **Data sources**: Industrial firewall logs, network flow data at IT/OT boundary. **App/TA** (typical add-on context): Firewall TAs, network flow data. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: network; **sourcetype**: pan:traffic. If that sourcetype is not mentioned in Data sources, double-check parsing or update the documentation to match the feed you actually ingest.

**Pipeline walkthrough**

• Scopes the data: index=network, sourcetype="pan:traffic". Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Filters the current rows with `where action="allow"` — typically the threshold or rule expression for this monitoring goal.
• `stats` rolls up events into metrics; results are split **by src, dest, dest_port, app** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
