# UC-14.7.3 · Edge-to-Splunk Data Pipeline Latency

## Description

End-to-end latency distinguishes edge capture delays from Splunk indexing backlog; spikes indicate broker saturation, HEC backpressure, or clock skew.

## Value

End-to-end latency distinguishes edge capture delays from Splunk indexing backlog; spikes indicate broker saturation, HEC backpressure, or clock skew.

## Implementation

Configure Litmus to stamp each tag event with capture time in epoch. Keep NTP synchronized on Litmus and Splunk. Alert when p95 exceeds acceptable thresholds.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Litmus Edge (Splunk HEC connector).
• Ensure the following data sources are available: `index=ot` `sourcetype="litmus:tag"` with `edge_timestamp` in JSON payload.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Configure Litmus to stamp each tag event with capture time in epoch. Keep NTP synchronized on Litmus and Splunk. Alert when p95 exceeds acceptable thresholds.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=ot sourcetype="litmus:tag"
| eval edge_sec=if(edge_timestamp>1e12, edge_timestamp/1000, edge_timestamp)
| where isnotnull(edge_timestamp) AND edge_sec>0
| eval latency_ms=abs(_time-edge_sec)*1000
| bin _time span=5m
| stats perc95(latency_ms) as p95_ms, avg(latency_ms) as avg_ms by gateway_id, _time
| where p95_ms > 5000
```
