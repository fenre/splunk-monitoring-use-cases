# UC-16.2.16 · Stale CI Discovery Freshness

## Description

Operational CIs without recent discovery updates inflate false confidence during impact analysis; freshness trending supports CMDB governance and ITIL configuration management objectives.

## Value

Operational CIs without recent discovery updates inflate false confidence during impact analysis; freshness trending supports CMDB governance and ITIL configuration management objectives.

## Implementation

(1) Tune the 90-day threshold to your discovery schedule; (2) Exclude classes intentionally not discovered (e.g., logical groups) via `where ci_class!=...`; (3) Publish a weekly remediation list to CMDB owners with `stale_days` for prioritization.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: `Splunk_TA_snow`.
• Ensure the following data sources are available: `index=itsm` `sourcetype="snow:cmdb_ci"` (`last_discovered`, `sys_updated_on`, `operational_status`, `name`, `ci_class`).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Tune the 90-day threshold to your discovery schedule; (2) Exclude classes intentionally not discovered (e.g., logical groups) via `where ci_class!=...`; (3) Publish a weekly remediation list to CMDB owners with `stale_days` for prioritization.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=itsm sourcetype="snow:cmdb_ci" operational_status="operational" earliest=-1d
| eval last_seen=coalesce(last_discovered, sys_updated_on)
| where isnotnull(last_seen) AND last_seen < relative_time(now(),"-90d@d")
| eval stale_days=round((now()-last_seen)/86400,0)
| stats count as stale_cis by ci_class, support_group
| sort -stale_cis
```
