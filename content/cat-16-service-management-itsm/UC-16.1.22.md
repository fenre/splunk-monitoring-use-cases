# UC-16.1.22 · Escalation Path Audit

## Description

Audits whether incidents followed the documented escalation chain (L1→L2→vendor) for compliance and training.

## Value

Audits whether incidents followed the documented escalation chain (L1→L2→vendor) for compliance and training.

## Implementation

Build assignment hop strings from audit `old_value`/`new_value`. Flag incidents with excessive hops for P1/P2 (policy threshold). Optionally `lookup escalation_policy.csv` with first/last hop pairs for stricter audits. Adjust field names to match `sys_audit` extractions.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: `Splunk_TA_snow`.
• Ensure the following data sources are available: Incident history / audit (`sourcetype=snow:incident:audit`).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Build assignment hop strings from audit `old_value`/`new_value`. Flag incidents with excessive hops for P1/P2 (policy threshold). Optionally `lookup escalation_policy.csv` with first/last hop pairs for stricter audits. Adjust field names to match `sys_audit` extractions.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=itsm sourcetype="snow:incident:audit" field_name="assignment_group"
| eval from_group=old_value, to_group=new_value
| eval step=from_group."->".to_group
| stats values(step) as escalation_path, count as hops by number
| join max=1 number [ search index=itsm sourcetype="snow:incident" priority IN (1,2) earliest=-90d | fields number, priority ]
| where hops>4
| table number, priority, escalation_path, hops
```
