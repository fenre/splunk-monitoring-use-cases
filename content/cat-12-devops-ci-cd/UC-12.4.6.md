# UC-12.4.6 · Pipeline Failure Root Cause Trending

## Description

Recurring failure causes (e.g., flaky tests, env issues) slow delivery. Trending root causes supports targeted remediation and stability.

## Value

Recurring failure causes (e.g., flaky tests, env issues) slow delivery. Trending root causes supports targeted remediation and stability.

## Implementation

Parse failure messages and stack traces from CI logs. Classify by cause (test, env, dependency, timeout). Alert on spike in a specific cause. Report on top failure reasons by job and week.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Jenkins/GitHub Actions/Azure DevOps TAs.
• Ensure the following data sources are available: Pipeline run logs, failure reasons, stage outcomes.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Parse failure messages and stack traces from CI logs. Classify by cause (test, env, dependency, timeout). Alert on spike in a specific cause. Report on top failure reasons by job and week.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=cicd sourcetype="jenkins:build"
| where result="FAILURE"
| rex field=message "(?<cause>Timeout|OOM|Connection refused|AssertionError|dependency)"
| stats count by cause, job_name
| sort -count
```

Understanding this SPL

**Pipeline walkthrough**

• Scopes the data: index=cicd, sourcetype="jenkins:build". Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Filters the current rows with `where result="FAILURE"` — typically the threshold or rule expression for this monitoring goal.
• Extracts fields with `rex` (regular expression).
• `stats` rolls up events into metrics; results are split **by cause, job_name** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
