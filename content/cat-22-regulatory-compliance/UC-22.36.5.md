# UC-22.36.5 · DSAR request-type mix anomaly — zero-deletion skew indicating broken workflow

## Description

Detects statistically anomalous drift in the distribution of DSAR request types (access / rectification / erasure / portability / opt-out). A sudden zero-erasure week across a large estate is usually a symptom of a broken UX flow rather than genuine user choice.

## Value

Makes the common 'silent bug in the privacy portal' failure mode visible without waiting for a complaint.

## Implementation

(1) Ensure DSAR events emit a canonical request_type from an allow-list; (2) run weekly; (3) on fire, open a DPO + engineering review ticket. Indexes required: `index=dsar`; Sourcetypes: `sourcetype=onetrust:dsar`, `sourcetype=servicenow:privacy`.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=dsar sourcetype=onetrust:dsar OR sourcetype=servicenow:privacy earliest=-84d
| bin _time span=7d
| stats count by _time request_type
| eventstats sum(count) as total by _time
| eval ratio=count/total
| stats avg(ratio) as baseline, stdev(ratio) as baseline_sd latest(ratio) as current by request_type
| eval drift=abs(current-baseline)
| where drift>(3*baseline_sd) AND baseline_sd>0
| table request_type baseline current drift baseline_sd
```
