# UC-22.15.38 · Embedded PLC Audit Log Forwarding Gaps

## Description

Identifies PLCs that stopped emitting security audit events to Splunk compared to a seven-day baseline.

## Value

Gives the responsible team auditor-ready evidence for IEC 62443 compliance by automating regulatory compliance evidence using indexes ot_eng and sourcetypes plc:audit. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Enable syslog/SNMP trap from PLC if supported; (2) UF on gateway concentrator; (3) alert on 24h silence; (4) spares swap procedure. Indexes required: `index=ot_eng`; Sourcetypes: `sourcetype=plc:audit`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: `metadata` or custom heartbeat.
• Ensure the following data sources are available: `index=ot_eng` per-controller heartbeat.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Enable syslog/SNMP trap from PLC if supported; (2) UF on gateway concentrator; (3) alert on 24h silence; (4) spares swap procedure.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=ot_eng sourcetype="plc:audit" earliest=-14d
| bin _time span=1d
| stats count by _time, controller
| eventstats avg(count) as baseline by controller
| where _time>relative_time(now(),"-3d@d") AND count < baseline*0.2
| table _time, controller, count, baseline
```

Understanding this SPL

**Pipeline walkthrough**

• Scopes the data: index=ot_eng, sourcetype="plc:audit", time bounds. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Discretizes time or numeric ranges with `bin`/`bucket`.
• `stats` aggregates events by **_time, controller**, producing one row per unique combination for trend analysis and threshold evaluation.
• `eventstats` rolls up events into metrics; results are split **by controller** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Filters the current rows with `where _time>relative_time(now(),"-3d@d") AND count < baseline*0.2` — typically the threshold or rule expression for this monitoring goal.

Enable Data Model Acceleration (and metric indexes for `mstats`) for the models or datasets referenced above; otherwise `tstats`/`mstats` may return no results from summaries.

Output fields: _time, baseline, controller, count.
