# UC-22.11.16 · Data Retention Job Failures for Cardholder Data Stores

## Description

Tracks scheduled purge and truncation jobs for databases that held PAN so retention limits are enforced operationally, not only on paper.

## Value

Gives the responsible team auditor-ready evidence for PCI DSS compliance by automating regulatory compliance evidence using indexes app, linux and sourcetypes batch:retention_job, sqlagent:job. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Instrument retention scripts to emit JSON lines with `job_name` and `status`. (2) Align job list with data retention standard table. (3) Page on two consecutive failures. (4) Attach weekly success rate to compliance dashboard. Indexes required: `index=app`, `index=linux`; Sourcetypes: `sourcetype=batch:retention_job`, `sourcetype=sqlagent:job`.

## Detailed Implementation

Prerequisites

• Ensure the following data sources are available: `index=app` `sourcetype="sqlagent:job"` or `index=linux` cron logs, `sourcetype="batch:retention_job"`.

Step 1 — Configure data collection
(1) Instrument retention scripts to emit JSON lines with `job_name` and `status`. (2) Align job list with data retention standard table. (3) Page on two consecutive failures. (4) Attach weekly success rate to compliance dashboard.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=app sourcetype="batch:retention_job" earliest=-7d
| where match(job_name,"(?i)pci|pan|retention|purge|truncate")
| eval success=if(match(status,"(?i)success|completed"),1,0)
| stats latest(_time) as last_run, sum(success) as ok_runs, count as total_runs by host, job_name
| eval ok_rate=if(total_runs>0, round(100*ok_runs/total_runs,2), null())
| where ok_rate<100 OR total_runs=0
| sort job_name
```
