# UC-22.7.43 · Executive Paging Lag After Sev-1 Playbook Start

## Description

Verifies paging actions fired within policy window after high-severity SOAR playbook start.

## Value

Verifies paging actions fired within policy window after high-severity SOAR playbook start.

## Implementation

(1) Standardize severity; (2) test quarterly; (3) document comms tree.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk SOAR, PagerDuty.
• Ensure the following data sources are available: `phantom:action_run`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Standardize severity; (2) test quarterly; (3) document comms tree.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=soar sourcetype=phantom:action_run earliest=-90d action="send pager"
| stats earliest(_time) as page_time by playbook_run_id
| join playbook_run_id [| search index=soar sourcetype=phantom:playbook_run severity=1 | rename _time as pb_time | fields playbook_run_id, pb_time]
| eval lag_sec=page_time-pb_time
| where lag_sec>1800
```

Understanding this SPL

• Scopes the data: index=soar, sourcetype=phantom:action_run, time bounds. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `stats` rolls up events into metrics; results are split **by playbook_run_id** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).

• Filters the current rows with `where lag_sec>1800` — typically the threshold or rule expression for this monitoring goal.
