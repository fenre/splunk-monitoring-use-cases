# UC-22.11.19 · Sensitive Authentication Data (SAD) in Auth Broker Logs

## Description

Detects track data, CVC, or PIN block patterns in authentication or tokenization service logs so SAD is not retained after authorization, per stored data prohibitions.

## Value

Gives the responsible team auditor-ready evidence for PCI DSS compliance by automating regulatory compliance evidence using indexes app, stream and sourcetypes oauth:gateway:json. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Restrict search to security admin role. (2) Integrate with DLP blocking on the log path. (3) Mandatory incident for any positive hit. (4) Legal/privacy review of regex scope annually. Indexes required: `index=app`, `index=stream`; Sourcetypes: `sourcetype=oauth:gateway:json`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security (263), Splunk Stream (1809).
• Ensure the following data sources are available: `index=app` `sourcetype="oauth:gateway:json"`; `index=stream` TLS payloads only where policy allows mirroring metadata.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=app sourcetype="oauth:gateway:json" earliest=-24h
| where match(_raw,"(?i)track[12]|;\\d{15,}|%B\\d{15,}|pin\\s*block|cvc2?\\s*[:=]\\s*\\d{3,4}")
| rex max_match=0 field=_raw "(?i)(?<redacted>track[12][^\
]{0,40})"
| stats count by host, endpoint, ip
| sort - count
```
