# 22.13.47 — Data Preservation During Recovery Activities

**Criticality:** critical | **Difficulty:** advanced | **Status:** N/A

## Description

We monitor mass deletes or index resets during declared recovery windows that could destroy forensic or compliance evidence.

## Value

Gives the responsible team auditor-ready evidence for NERC CIP compliance by automating regulatory compliance evidence using indexes _audit, itsm and sourcetypes maint_window. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=_audit (action=delete OR match(action,"(?i)clean")) earliest=-7d
| lookup bes_splunk_indexes.csv index OUTPUT bes_evidence_index
| where bes_evidence_index="true"
| join type=left user [
    search index=itsm sourcetype=maint_window earliest=-14d
    | eval mw=1
    | fields user mw
  ]
| where isnull(mw)
| table _time, user, action, index, object
```

## Implementation

(1) Tag evidence-bearing indexes in lookup. (2) Require maintenance window tickets for bulk deletes. (3) Page IR and compliance on hits. (4) Pair with bucket freeze procedures. Indexes required: `index=_audit`, `index=itsm`; Sourcetypes: `sourcetype=maint_window`.

## Visualization

Table (events), Timeline, Single value (violations). Key fields: `action`, `index`, `mw`, `object`, `user`.

## Compliance

- **NERC CIP** current: CIP-009-6 R4 Part 4.1 (partial) — Uplifted from contributing to partial: UC provides meaningful monitoring evidence for CIP-009-6 R4 Part 4.1.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
