# UC-22.5.25 · MiFID II Transaction Reporting — End-to-End Latency from Execution to ARM Accept Acknowledgment

## Description

Timeliness reduces backlogs that cause late reports. Measuring latency from internal execution capture to ARM acceptance ACK supports operational resilience of the reporting chain beyond static “file sent” checks.

## Value

Timeliness reduces backlogs that cause late reports. Measuring latency from internal execution capture to ARM acceptance ACK supports operational resilience of the reporting chain beyond static “file sent” checks.

## Implementation

(1) Clock sync NTP monitoring separately; (2) tune SLA ms by instrument; (3) alert reporting ops; (4) dashboard by venue session; (5) retain for audit trail.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: OMS + ARM via correlated trace IDs in HEC.
• Ensure the following data sources are available: `index=trading` `sourcetype="oms:exec"` (exec_id, exec_ts); `index=reg` `sourcetype="arm:ack"` (exec_id, ack_ts, status).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=trading sourcetype="oms:exec" earliest=-7d
| eval et=strptime(exec_ts,"%Y-%m-%dT%H:%M:%SZ")
| join type=left exec_id [
    search index=reg sourcetype="arm:ack" status="ACCEPTED" earliest=-7d
    | eval at=strptime(ack_ts,"%Y-%m-%dT%H:%M:%SZ")
    | stats min(at) as first_ack by exec_id
  ]
| eval latency_ms=(first_ack-et)*1000
| where latency_ms>5000 OR isnull(first_ack)
| stats perc95(latency_ms) as p95_ms, count as breaches
```
