# UC-22.4.25 ┬Ę CCPA/CPRA ŌĆö Correction/Deletion Verification ŌĆö Search Index and Cache Purge Lag After Correction Request

## Description

Corrections must appear consistently across channels. Measuring lag between CRM correction timestamps and search-index update events (or CDN cache purge) demonstrates you catch stale copies that could frustrate consumer expectations under ┬¦1798.106.

## Value

Gives the responsible team auditor-ready evidence for CCPA/CPRA compliance by automating regulatory compliance evidence using indexes crm, search and sourcetypes crm:profile_update, search:indexer_audit. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Align `consumer_id` to search document keys; (2) tune `24` hours to product SLA; (3) include CDN purge events in union if indexed separately; (4) customer support notified on breach; (5) quarterly control testing evidence. Indexes required: `index=crm`, `index=search`; Sourcetypes: `sourcetype=crm:profile_update`, `sourcetype=search:indexer_audit`.

Step 2 ŌĆö Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=crm sourcetype="crm:profile_update" corrected_field="mailing_address" earliest=-30d
| eval ct=strptime(corrected_ts,"%Y-%m-%dT%H:%M:%SZ")
| join type=left consumer_id [
    search index=search sourcetype="search:indexer_audit" index_operation IN ("UPSERT","REINDEX_DOC") earliest=-30d
    | rename _time as index_time
    | fields consumer_id, index_time
  ]
| eval lag_h=(index_time-ct)/3600
| where lag_h>24 OR isnull(index_time)
| table consumer_id, ct, index_time, lag_h
```
