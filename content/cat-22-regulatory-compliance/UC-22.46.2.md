# UC-22.46.2 · Phishing simulation efficacy — click-rate trend and repeat-clicker detection

## Description

Tracks efficacy: the programme is green only if click-rate stays within baseline and repeat-clickers are followed up. Drift above threshold is a programme-level finding.

## Value

Turns the awareness programme into a measurable control, not an annual training badge.

## Implementation

(1) Ingest campaign events from the phishing-sim platform; (2) compute baselines per quarter; (3) run monthly; (4) deliver repeat-clicker lists to HR for targeted training. Indexes required: `index=phishing`; Sourcetypes: `sourcetype=knowbe4:campaign`, `sourcetype=proofpoint:psa`.

Step 1 — Configure data collection

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=phishing sourcetype=knowbe4:campaign OR sourcetype=proofpoint:psa earliest=-180d
| bin _time span=30d
| stats count(eval(action="clicked")) as clicks count as total by _time
| eval click_rate=round(clicks/total*100,2)
| stats last(click_rate) as current avg(click_rate) as baseline
| eval drift=current-baseline
| where drift>3
| appendpipe [search index=phishing action=clicked earliest=-180d | stats dc(campaign_id) as clicks by employee | where clicks>=3]
```

## Known False Positives

Newly-onboarded employees inflate click-rate; compute baseline excluding the first 30 days post-start.
