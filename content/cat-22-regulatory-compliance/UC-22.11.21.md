# UC-22.11.21 ¬∑ Cryptographic Key Rotation and Custodian Acknowledgement Trail

## Description

Tracks scheduled key rotation events and custodian acknowledgements so the organization can demonstrate keys are replaced on policy cadence and custodianship is documented.

## Value

Tracks scheduled key rotation events and custodian acknowledgements so the organization can demonstrate keys are replaced on policy cadence and custodianship is documented.

## Implementation

(1) Populate `expected_days` per key from the cryptographic procedures. (2) Align `span` to rotation policy (annual vs quarterly). (3) Append manual HSM rotation logs via HEC with same `key_id`. (4) Archive quarterly attestation PDFs externally.

## Detailed Implementation

Prerequisites

‚ÄĘ Ensure the following data sources are available: `index=aws` `sourcetype="aws:cloudtrail"` KMS `RotateKey`; `index=sec` `sourcetype="vault:audit"`; `index=itsm` CAB tasks.
‚ÄĘ For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 ‚ÄĒ Configure data collection
(1) Populate `expected_days` per key from the cryptographic procedures. (2) Align `span` to rotation policy (annual vs quarterly). (3) Append manual HSM rotation logs via HEC with same `key_id`. (4) Archive quarterly attestation PDFs externally.

Step 2 ‚ÄĒ Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
(index=aws sourcetype="aws:cloudtrail" eventName="RotateKey" earliest=-365d)
 OR (index=sec sourcetype="vault:audit" operation="rotate" earliest=-365d)
| eval key_id=coalesce(requestParameters.keyId, key_id)
| bin _time span=90d
| stats count as rotations by _time, key_id
| lookup pci_key_rotation_policy.csv key_id OUTPUT expected_days
| eval expected_rotations=365/expected_days
| where rotations<1
| table _time, key_id, rotations, expected_rotations
```
