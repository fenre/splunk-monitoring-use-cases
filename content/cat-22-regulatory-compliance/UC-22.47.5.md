# UC-22.47.5 · Continuous control monitoring anomaly — failure-rate trending up

## Description

Surfaces controls whose failure rate is trending up — the most reliable early-warning signal that a control is drifting.

## Value

Gives internal audit a quantitative view of control health rather than a point-in-time pass/fail.

## Implementation

(1) Instrument CCM per control; (2) run daily; (3) open a deep-dive ticket for anomalies. Indexes required: `index=grc`; Sourcetypes: `sourcetype=ccm:execution`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform.
• Ensure the following data sources are available: GRC continuous-control-monitoring events (pass/fail per execution)..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Instrument CCM per control; (2) run daily; (3) open a deep-dive ticket for anomalies.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=grc sourcetype=ccm:execution earliest=-90d
| bin _time span=1d
| stats count(eval(result="fail")) as fails count as total by _time control_id
| eval fail_rate=fails/total
| eventstats avg(fail_rate) as avg7d by control_id
| stats avg(fail_rate) as baseline stdev(fail_rate) as baseline_sd latest(fail_rate) as current latest(avg7d) as avg7d by control_id
| where avg7d > baseline+3*baseline_sd
| table control_id baseline baseline_sd avg7d current
```
