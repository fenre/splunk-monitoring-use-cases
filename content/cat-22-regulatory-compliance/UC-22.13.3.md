# UC-22.13.3 · ESP Boundary Change Detection via Firewall Configuration Events

## Description

We tie ESP security rule, zone, and interface edits to change records so boundary changes to BES Cyber Systems are visible and attributable.

## Value

We tie ESP security rule, zone, and interface edits to change records so boundary changes to BES Cyber Systems are visible and attributable.

## Implementation

(1) Forward Panorama or device config logs. (2) Map changed objects to ESP segments in `esp_object_inventory`. (3) Require ITSM linkage for production commits. (4) Weekly compliance dashboard export.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Add-on for Palo Alto Networks (2757), Splunk Enterprise Security (263).
• Ensure the following data sources are available: `index=network` `sourcetype=pan:config` OR `sourcetype=pan:system`, `index=itsm` `sourcetype=chg:firewall`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Forward Panorama or device config logs. (2) Map changed objects to ESP segments in `esp_object_inventory`. (3) Require ITSM linkage for production commits. (4) Weekly compliance dashboard export.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=network (sourcetype=pan:config OR sourcetype=pan:system) earliest=-7d
| where match(object,"(?i)security|zone|interface") OR match(cmd,"(?i)commit")
| lookup esp_object_inventory object OUTPUT esp_segment_id
| join type=left max=0 object [
    search index=itsm sourcetype=chg:firewall earliest=-30d
    | stats latest(approved) as approved latest(change_id) as change_id by firewall_object
    | rename firewall_object as object
  ]
| where approved!="true" OR isnull(change_id)
| table _time, admin, object, esp_segment_id, change_id, approved
```
