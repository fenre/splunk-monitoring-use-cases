# UC-22.36.4 · DSAR identity-verification friction — failed-verification anomaly

## Description

Surfaces data-subjects whose right-exercise attempts repeatedly fail verification, which — aggregated over time — is a reliable indicator either of a broken verification UX or of a deliberate design that frustrates rights. Either interpretation is a DPO priority.

## Value

Takes the single most common enforcement narrative ('the verification process itself denied the right') and turns it into a measurable metric.

## Implementation

(1) Ensure DSAR verification events emit subject_token and result; (2) run daily; (3) feed findings to a DPO review dashboard; (4) after a fixed threshold, escalate via SOAR for legal review. Indexes required: `index=dsar`; Sourcetypes: `sourcetype=onetrust:dsar`, `sourcetype=servicenow:privacy`.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=dsar sourcetype=onetrust:dsar OR sourcetype=servicenow:privacy action=verification earliest=-14d
| eval pass=if(result="success",1,0), fail=if(result="failure",1,0)
| stats sum(pass) as passes sum(fail) as failures values(ticket_id) as ticket_ids by subject_token
| where failures>=3 AND passes<1
| table subject_token failures ticket_ids
```

## Known False Positives

Genuine impersonation attempts will (correctly) produce multiple failures for the same subject_token; pair with IdP risk signals before escalating.
