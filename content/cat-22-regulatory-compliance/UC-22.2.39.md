# 22.2.39 — NIS2 OT Incident Detection — Process and Protocol Anomalies

**Criticality:** critical | **Difficulty:** advanced | **Status:** verified

## Description

OT incidents may manifest as Modbus function anomalies, unusual OPC-UA browse depth, or historian gaps rather than Windows malware. This use case baselines industrial protocol traffic per cell line and alerts on deviations that suggest manipulation or sensor spoofing.

## Value

Gives the responsible team auditor-ready evidence for NIS2 compliance by automating regulatory compliance evidence using indexes ot_protocol and sourcetypes modbus:session. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=ot_protocol sourcetype="modbus:session" earliest=-7d
| stats count dc(function_code) as fn_variety avg(register_span) as avg_span by plc_name cell_line
| lookup nis2_ot_baseline.csv cell_line OUTPUT expected_fn_variety expected_span
| eval anomaly=if(fn_variety > expected_fn_variety*2 OR avg_span > expected_span*3, 1, 0)
| where anomaly=1
| table _time, plc_name, cell_line, fn_variety, expected_fn_variety, avg_span, expected_span
```

## Implementation

(1) Build baselines per production cell during golden runs; (2) ingest Modbus/OPC-UA metadata without disrupting real-time control; (3) tune thresholds with engineering; (4) route alerts to OT SOC; (5) feed confirmed incidents into UC-22.2.1/UC-22.2.33 reporting paths. Indexes required: `index=ot_protocol`; Sourcetypes: `sourcetype=modbus:session`; Lookups: `nis2_ot_baseline`, `nis2_ot_baseline.csv`.

## Visualization

Line chart (function code diversity), Table (anomalous cells), Single value (anomaly count), Heatmap (by shift).

## Compliance

- **NIS2** Directive (EU) 2022/2555: Art.21(2)(f) (partial) — Uplifted from contributing: SPL directly addresses Art.21(2)(f) requirements with specific, actionable monitoring.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
