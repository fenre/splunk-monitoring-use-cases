# 22.2.15 — NIS2 Secure System Acquisition and Development Lifecycle

**Criticality:** high | **Difficulty:** intermediate | **Status:** N/A

## Description

Article 21(2)(e) extends beyond vulnerability management to require security in system acquisition, development, and maintenance lifecycles. This use case monitors CI/CD pipeline security gates, tracks code scanning results, and validates that change management processes include security review — demonstrating that security is embedded in the development lifecycle rather than bolted on after deployment.

## Value

Gives the responsible team auditor-ready evidence for NIS2 compliance by automating regulatory compliance evidence using indexes codescan, devops, itsm and sourcetypes github:webhook, gitlab:pipeline, jenkins:build. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=devops sourcetype IN ("github:webhook","gitlab:pipeline","jenkins:build") earliest=-30d
| eval has_security_scan=if(match(lower(_raw),"(?i)sast|sca|snyk|sonar|trivy|semgrep|security.scan|code.scan"), 1, 0)
| eval deployment=if(match(lower(_raw),"(?i)deploy|release|prod|production"), 1, 0)
| stats count as total_builds, sum(has_security_scan) as scanned_builds, sum(deployment) as deployments by repository
| eval scan_coverage=round(100*scanned_builds/total_builds, 1)
| where scan_coverage < 80 OR (deployments > 0 AND scanned_builds = 0)
| sort scan_coverage
| table repository, total_builds, scanned_builds, scan_coverage, deployments
```

## Implementation

(1) Forward CI/CD pipeline events to Splunk via webhooks or HEC; (2) define minimum security gate requirements (SAST scan, dependency check, container image scan); (3) alert on deployments to production without security scan evidence; (4) track scan coverage percentage as a NIS2 KPI; (5) correlate with change management tickets to verify security review sign-off. Indexes required: `index=codescan`, `index=devops`, `index=itsm`; Sourcetypes: `sourcetype=github:webhook`, `sourcetype=gitlab:pipeline`, `sourcetype=jenkins:build`.

## Visualization

Table (repos without scans), Bar chart (scan coverage by repo), Single value (overall scan coverage %), Line chart (coverage trend). Key fields: `deployment`, `deployments`, `has_security_scan`, `repository`, `scan_coverage`, `scanned_builds`.

## Compliance

- **NIS2** Directive (EU) 2022/2555: Art.21(2)(e) (partial) — Uplifted from contributing: SPL directly addresses Art.21(2)(e) requirements with specific, actionable monitoring.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
