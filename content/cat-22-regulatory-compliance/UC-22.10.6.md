# UC-22.10.6 · Security Awareness & Training — Phish Simulation Failure to ePHI Risk

## Description

Links repeated phishing test failures to subsequent risky ePHI handling patterns so awareness gaps become targeted coaching, not checkbox training.

## Value

Links repeated phishing test failures to subsequent risky ePHI handling patterns so awareness gaps become targeted coaching, not checkbox training.

## Implementation

(1) Prefer `lookup` + subsearch limits for scale; (2) de-identify in dashboards; (3) HR/privacy joint review; (4) document coaching in LMS export.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security (263), Splunk Add-on for Microsoft Office 365 (4055).
• Ensure the following data sources are available: `index=security_awareness` HEC (`campaign_id`, `user`, `result`), `index=o365` `sourcetype=ms:o365:management` (DLP/alerts), `index=epic` chart exports.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Prefer `lookup` + subsearch limits for scale; (2) de-identify in dashboards; (3) HR/privacy joint review; (4) document coaching in LMS export.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=security_awareness sourcetype="phish:sim" result="Clicked" earliest=-90d
| stats count as fail_clicks by user
| where fail_clicks>=3
| map maxsearches=50 search="index=epic sourcetype=\"epic:audit\" USER_ID=\"$user$\" AccessType IN (\"Report\",\"Export\") earliest=-90d | stats count as risky_epic_events by USER_ID"
| sort - risky_epic_events
```

**Pipeline walkthrough**

• Scopes the data: index=security_awareness, sourcetype="phish:sim", time bounds. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `stats` rolls up events into metrics; results are split **by user** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).

• Runs a templated search per row with `map`.
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
