# UC-22.22.20 · Settlement Reconciliation Exceptions (Acquirer vs Issuer)

## Description

Provides transaction-level integrity and monitoring evidence aligned with PSD2 operational and security expectations for ASPSPs and payment processors.

## Value

Provides transaction-level integrity and monitoring evidence aligned with PSD2 operational and security expectations for ASPSPs and payment processors.

## Implementation

(1) Normalize field extractions and CIM tags; (2) schedule hourly or near-real-time; (3) route alerts to SOC/compliance queues; (4) export evidence packs for supervisory review.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk ITSI (1841), Splunk DB Connect (2686).
• Ensure the following data sources are available: `index=payments` `pay:settlement` — authorization_id, merchant_id, amount_eur, fx_rate, reversal_flag.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Normalize field extractions and CIM tags; (2) schedule hourly or near-real-time; (3) route alerts to SOC/compliance queues; (4) export evidence packs for supervisory review.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=payments sourcetype="pay:settlement" earliest=-24h
| eval txn_tag="settle"
| eval eur_equiv=coalesce(amount_eur, amount*fx_rate)
| stats count as txns, dc(authorization_id) as auth_dc, sum(eval(eur_equiv>100000)) as very_high by merchant_id, txn_tag, bin(_time,1h)
| eval dup_suspect=if(txns>auth_dc AND txn_tag="dup", txns-auth_dc, 0)
| eval uc_id="22.22.20"
| where very_high>0 OR dup_suspect>5 OR txn_tag="refund"
| sort - very_high
| head 200
```
