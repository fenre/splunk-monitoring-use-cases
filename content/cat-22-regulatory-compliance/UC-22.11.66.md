# UC-22.11.66 · Critical System Clock Skew Between Database and Application Tier

## Description

Compares `_time` deltas between paired application and database events sharing a `correlation_id` so forensic timelines for CHD access stay coherent across tiers.

## Value

Gives the responsible team auditor-ready evidence for PCI DSS compliance by automating regulatory compliance evidence using indexes app, db and sourcetypes oracle:audit, payment:api:json. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Require apps to propagate `correlation_id` into DB audit. (2) Tune threshold per transaction latency SLA. (3) Pair with NTP health UC. (4) Weekly ops review. Indexes required: `index=app`, `index=db`; Sourcetypes: `sourcetype=oracle:audit`, `sourcetype=payment:api:json`.

## Detailed Implementation

Step 1 — Configure data collection
(1) Require apps to propagate `correlation_id` into DB audit. (2) Tune threshold per transaction latency SLA. (3) Pair with NTP health UC. (4) Weekly ops review.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
(index=app sourcetype="payment:api:json" earliest=-1h) OR (index=db sourcetype="oracle:audit" earliest=-1h)
| where isnotnull(correlation_id)
| eval src=if(index="app","app","db")
| stats earliest(eval(if(src="app",_time,null()))) as t_app, earliest(eval(if(src="db",_time,null()))) as t_db by correlation_id
| eval skew_sec=abs(t_app-t_db)
| where skew_sec>5 AND isnotnull(t_app) AND isnotnull(t_db)
| stats perc95(skew_sec) as p95_skew, count by correlation_id
| where p95_skew>5
| sort - p95_skew
| head 100
```
