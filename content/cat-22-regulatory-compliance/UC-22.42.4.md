# UC-22.42.4 · CAB approval bypass — change pushed before scheduled window

## Description

Correlates every production apply event to its CR scheduled window and emits those that landed outside the declared window. Out-of-window changes that are not flagged as emergency are a primary ITGC deficiency.

## Value

Turns CAB approval from a calendar event into an operational control with per-change evidence.

## Implementation

(1) Force CI/CD pipelines to emit change_id; (2) ingest CR schedule data; (3) run hourly; (4) SOAR auto-freezes the pipeline on hit.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform.
• Ensure the following data sources are available: CI/CD pipeline apply events tagged with change_id, ServiceNow CR schedule data..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Force CI/CD pipelines to emit change_id; (2) ingest CR schedule data; (3) run hourly; (4) SOAR auto-freezes the pipeline on hit.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=change sourcetype=prod:apply earliest=-1h
| join change_id [search index=itsm sourcetype=servicenow:change | fields change_id window_start window_end approval_type]
| eval start_s=strptime(window_start,"%Y-%m-%dT%H:%M:%SZ"), end_s=strptime(window_end,"%Y-%m-%dT%H:%M:%SZ")
| where approval_type!="emergency" AND (_time<start_s OR _time>end_s)
| table _time change_id applied_by window_start window_end
```

## Known False Positives

Timezone-normalised window_start/window_end is critical; a TZ mismatch will falsely flag correct changes.
