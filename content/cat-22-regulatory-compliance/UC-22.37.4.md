# UC-22.37.4 · Purpose-limitation enforcement — processing not matching declared purpose

## Description

Observes access tokens carrying a declared processing purpose and compares that purpose to the declared purposes of the system being accessed. Any mismatch is evidence that the purpose-limitation principle is not being enforced.

## Value

Converts an otherwise-abstract regulatory principle into a continuously-measurable control with evidentiary value.

## Implementation

(1) Mint OAuth/OIDC tokens with a purpose claim; (2) publish purpose_catalogue.csv keyed by system_id; (3) run hourly; (4) alert DPO on mismatches and drive remediation via SOAR.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform.
• Ensure the following data sources are available: Application audit logs carrying the purpose-bearing access token, system-purpose catalogue in purpose_catalogue.csv..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Mint OAuth/OIDC tokens with a purpose claim; (2) publish purpose_catalogue.csv keyed by system_id; (3) run hourly; (4) alert DPO on mismatches and drive remediation via SOAR.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=app_activity earliest=-1h token_purpose=*
| lookup purpose_catalogue.csv system_id OUTPUT allowed_purposes
| eval ok=if(match(allowed_purposes,token_purpose),1,0)
| where ok=0
| stats count values(user) as users values(subject_token) as subjects by system_id token_purpose allowed_purposes
| table system_id token_purpose allowed_purposes count users subjects
```

## Known False Positives

Shared service accounts without per-request purpose context cannot be evaluated; filter them out via service_account_allowlist.csv.
