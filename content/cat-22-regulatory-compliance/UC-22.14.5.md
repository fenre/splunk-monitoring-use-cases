# UC-22.14.5 · Audit Review, Analysis, and Reporting for Privileged Users

## Description

Produces repeatable privileged-user activity summaries for periodic AU-6 audit review and management attestation.

## Value

Gives the responsible team auditor-ready evidence for NIST 800-53 compliance by automating regulatory compliance evidence using indexes sec. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Define `privileged=1` via `identities.csv` or ES asset priority; (2) replace macro with your CIM index macro; (3) schedule weekly PDF/CSV to GRC; (4) store signed exports for assessor sampling. Indexes required: `index=sec`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security (Splunkbase 263).
• Ensure the following data sources are available: ES assets/identities (optional), `index=sec` Authentication-tagged events.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Define `privileged=1` via `identities.csv` or ES asset priority; (2) replace macro with your CIM index macro; (3) schedule weekly PDF/CSV to GRC; (4) store signed exports for assessor sampling.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=sec earliest=-7d
| search `cim_Authentication_indexes` (user="*admin*" OR user="*svc_*" OR privileged=1)
| stats count by user, action, dest, app
| sort - count
| head 200
```

**Pipeline walkthrough**

• Scopes the data: index=sec, time bounds. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• Applies an explicit `search` filter to narrow the current result set.
• `stats` rolls up events into metrics; results are split **by user, action, dest, app** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
• Limits the number of rows with `head`.

Optional CIM / accelerated variant (same use case, normalized fields via Common Information Model):

```spl
| tstats summariesonly=t count from datamodel=Authentication.Authentication by Authentication.user, Authentication.action, Authentication.dest, Authentication.app | sort - count
```
