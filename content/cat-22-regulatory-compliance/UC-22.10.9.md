# UC-22.10.9 · Periodic Evaluation — Control Test Evidence Ingest

## Description

Centralizes periodic technical test results (MFA coverage, encryption checks) with timestamps for HIPAA periodic technical and non-technical evaluation evidence.

## Value

Centralizes periodic technical test results (MFA coverage, encryption checks) with timestamps for HIPAA periodic technical and non-technical evaluation evidence.

## Implementation

(1) Standardize `result` enum Pass/Fail/Partial; (2) join to asset inventory; (3) annual HIPAA evaluation deck exports from this dataset.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud (HEC), Splunk ITSI (1841).
• Ensure the following data sources are available: `index=grc` `sourcetype=hipaa:periodic_test` JSON lines from Qualys/Azure Policy/AWS Config exporters.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Standardize `result` enum Pass/Fail/Partial; (2) join to asset inventory; (3) annual HIPAA evaluation deck exports from this dataset.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=grc sourcetype="hipaa:periodic_test" earliest=-365d test_family="HIPAA_Technical"
| eval period_month=strftime(_time,"%Y-%m")
| stats latest(result) as last_result latest(_time) as last_run by control_id, period_month
| where last_result!="Pass"
| sort period_month, control_id
```

Understanding this SPL

**Periodic Evaluation — Control Test Evidence Ingest (§164.308(a)(8))** — Centralizes periodic technical test results (MFA coverage, encryption checks) with timestamps for HIPAA periodic technical and non-technical evaluation evidence.

Documented **Data sources**: `index=grc` `sourcetype=hipaa:periodic_test` JSON lines from Qualys/Azure Policy/AWS Config exporters. **App/TA** (typical add-on context): Splunk Enterprise / Splunk Cloud (HEC), Splunk ITSI (1841). The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: grc; **sourcetype**: hipaa:periodic_test. That sourcetype matches what this use case lists under Data sources.

**Pipeline walkthrough**

• Scopes the data: index=grc, sourcetype="hipaa:periodic_test", time bounds. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `eval` defines or adjusts **period_month** — often to normalize units, derive a ratio, or prepare for thresholds.
• `stats` rolls up events into metrics; results are split **by control_id, period_month** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Filters the current rows with `where last_result!="Pass"` — typically the threshold or rule expression for this monitoring goal.
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.
