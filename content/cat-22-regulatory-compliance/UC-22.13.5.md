# UC-22.13.5 · Security Policy Exception Register with Expiration Tracking

## Description

We prevent policy exceptions from silently exceeding approved dates so security management controls stay within delegated risk acceptance.

## Value

We prevent policy exceptions from silently exceeding approved dates so security management controls stay within delegated risk acceptance.

## Implementation

(1) Ingest the exception register via HEC or DB Connect on change. (2) Map `control_ref` to CIP-003 policies. (3) Alert at fourteen days and on expiration. (4) Require closure events to flip status.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security (263).
• Ensure the following data sources are available: `index=grc` `sourcetype=cip:policy_exception` (exception_id, control_ref, expires_on, status).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Ingest the exception register via HEC or DB Connect on change. (2) Map `control_ref` to CIP-003 policies. (3) Alert at fourteen days and on expiration. (4) Require closure events to flip status.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=grc sourcetype=cip:policy_exception status="active" earliest=-120d
| eval exp_epoch=strptime(expires_on,"%Y-%m-%d")
| eval expired=if(now()>exp_epoch,1,0)
| eval days_left=round((exp_epoch-now())/86400,1)
| where expired=1 OR days_left<14
| table exception_id, control_ref, owner, expires_on, expired, days_left
```
