# UC-22.43.1 · Critical vulnerability SLA tracker — unpatched 30+ days with exploited-in-the-wild indicator

## Description

Continuously measures how long critical or exploited-in-the-wild vulnerabilities have persisted. Breaches are surfaced immediately and feed the evidence pack with per-host details.

## Value

A missed critical patch is the #1 root cause of breaches — proving remediation timeliness is the matching audit control.

## Implementation

(1) Ingest scanner data into Splunk via TA; (2) consume CISA KEV list into exploited_in_wild.csv; (3) run daily; (4) integrate with SOAR to auto-open remediation.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform.
• Ensure the following data sources are available: See subcategory preamble..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Ingest scanner data into Splunk via TA; (2) consume CISA KEV list into exploited_in_wild.csv; (3) run daily; (4) integrate with SOAR to auto-open remediation.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=vuln sourcetype=tenable:vm OR sourcetype=qualys:vm OR sourcetype=rapid7:vm earliest=-60d cvss_v3_base>=9.0
| stats min(_time) as first_seen max(_time) as last_seen values(cve) as cves by host
| eval age_days=round((now()-first_seen)/86400,0)
| where age_days>30
| join type=outer host cves [search index=vuln action=remediated]
| where isnull(remediated_at)
| table host cves first_seen age_days
```
