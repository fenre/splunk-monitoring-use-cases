# 22.41.3 — Key rotation attestation — KMS/HSM rotation SLA tracker

**Criticality:** high | **Difficulty:** advanced | **Status:** community

## Description

Verifies every encryption key is rotated within its policy cryptoperiod. Overdue keys are highlighted and evidence is written to the compliance index for audit.

## Value

Gives the cryptographic custodian the ability to defend the 'our keys are rotated' claim with per-key evidence rather than a policy document.

## SPL

```spl
index=kms (sourcetype=aws:kms OR sourcetype=azure:keyvault OR sourcetype=gcp:kms) action=rotate
| stats max(_time) as last_rotation by key_id tenant
| lookup key_policy.csv key_id OUTPUT cryptoperiod_days key_type
| where key_type!="archive"
| eval age_days=round((now()-last_rotation)/86400,0)
| where age_days>cryptoperiod_days
| table tenant key_id last_rotation age_days cryptoperiod_days
```

## Implementation

(1) Ingest KMS audit logs; (2) maintain key_policy.csv; (3) run daily; (4) escalate via SOAR to the crypto custodian. Indexes required: `index=kms`; Sourcetypes: `sourcetype=aws:kms`, `sourcetype=azure:keyvault`, `sourcetype=gcp:kms`.

## Visualization

Table of overdue keys, scatter plot age vs cryptoperiod, single-value '% of keys within cryptoperiod'. Key fields: `age_days`, `cryptoperiod_days`, `key_id`, `last_rotation`, `tenant`.

## Compliance

- **PCI DSS** v4.0: 3.6 (full) — PCI 3.6 requires key rotation at end-of-cryptoperiod; the UC evidences compliance per key.
- **NIST 800-53** Rev. 5: SC-13 (full) — SC-13 key management encompasses rotation.
- **DORA** Regulation (EU) 2022/2554: Art.9 (full) — Uplifted to full: SPL provides comprehensive, standalone evidence for Art.9 with specific data sources and actionable thresholds.

## Known False Positives

Keys under a temporary freeze (key_status=frozen) for investigation should be excluded via a status filter.
