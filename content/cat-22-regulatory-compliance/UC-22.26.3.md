# UC-22.26.3 · NSM reporting compliance

## Description

Measures incident queue aging against internal NSM-aligned reporting deadlines including evidence attachments.

## Value

Gives the responsible team auditor-ready evidence for NO Sikkerhetsloven compliance by automating regulatory compliance evidence. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Maintain supporting lookups (classified hosts, sectors, transfers) aligned to Norwegian regulatory language. (2) Schedule searches with Europe/Oslo time semantics for reporting cutoffs. (3) Route ES notables to the correct national security, OT, or privacy queues. (4) Quarterly false-positive tuning with HR, operations, and legal stakeholders.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security (263), Splunk Add-on for Microsoft Windows (742), Splunk Add-on for Unix and Linux (833), Splunk OT Security Add-on (5151), Splunk ITSI (1841), Splunk Common Information Model Add-on (1621).
• Ensure the following data sources are available: Windows Security Event Logs, network infrastructure logs, OT/ICS system logs, power grid SCADA, petroleum control systems, government system audit trails, identity management systems, security incident reports.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Maintain supporting lookups (classified hosts, sectors, transfers) aligned to Norwegian regulatory language. (2) Schedule searches with Europe/Oslo time semantics for reporting cutoffs. (3) Route ES notables to the correct national security, OT, or privacy queues. (4) Quarterly false-positive tuning with HR, operations, and legal stakeholders.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
`notable` (tag="nsm_reportable" OR rule_name="*national*security*") earliest=-30d
| eval detected=_time
| eval reported=strptime(reported_authority_at,"%Y-%m-%d %H:%M:%S")
| eval hours=round((reported-detected)/3600,2)
| eval breach=if(isnull(reported) AND (now()-detected)>86400,1,if(hours>24,1,0))
| table _time, rule_name, owner, status, hours, breach
| sort - breach, - hours
```
