# 22.13.52 — Vulnerability Assessment After Material Configuration Changes

**Criticality:** high | **Difficulty:** intermediate | **Status:** N/A

## Description

We find material firewall or OS changes without a subsequent vulnerability scan within the required follow-up window.

## Value

Gives the responsible team auditor-ready evidence for NERC CIP compliance by automating regulatory compliance evidence using indexes itsm, vuln and sourcetypes chg:network, tenable:scans. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=itsm sourcetype=chg:network material=1 earliest=-35d
| eval chg_time=strptime(closed_at,"%Y-%m-%d %H:%M:%S")
| rename number as change_id
| join type=left change_id [
    search index=vuln sourcetype=tenable:scans earliest=-35d
    | rename change_ticket as change_id
    | stats latest(finished_at) as scan_done by change_id
  ]
| eval scan_epoch=strptime(scan_done,"%Y-%m-%d %H:%M:%S")
| where isnull(scan_epoch) OR (scan_epoch-chg_time)>1209600
| table change_id, closed_at, scan_done
```

## CIM SPL

```spl
| tstats summariesonly=t count from datamodel=Vulnerabilities.Vulnerabilities by Vulnerabilities.severity, Vulnerabilities.signature, Vulnerabilities.dest | sort - count
```

## Implementation

(1) Flag material changes in ITSM. (2) Require scanner to tag `change_ticket`. (3) Alert at fourteen days without scan. (4) Document risk acceptance if delayed. Indexes required: `index=itsm`, `index=vuln`; Sourcetypes: `sourcetype=chg:network`, `sourcetype=tenable:scans`.

## Visualization

Table (missing scans), Timeline, Single value (open gaps). Key fields: `change_id`, `chg_time`, `closed_at`, `scan_done`, `scan_epoch`.

## Compliance

- **NERC CIP** current: CIP-010-4 R4 Part 4.1 (partial) — Uplifted from contributing to partial: UC provides meaningful monitoring evidence for CIP-010-4 R4 Part 4.1.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
