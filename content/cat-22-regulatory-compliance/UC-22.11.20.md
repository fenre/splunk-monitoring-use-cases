# UC-22.11.20 · Hash and Truncation Method Changes on Tokenization Database

## Description

Alerts when DDL or configuration changes alter hashing algorithms or truncation rules applied to account data, preserving integrity of non-reversible rendering approaches approved by the assessor.

## Value

Alerts when DDL or configuration changes alter hashing algorithms or truncation rules applied to account data, preserving integrity of non-reversible rendering approaches approved by the assessor.

## Implementation

(1) Enable fine-grained audit on token vault schema. (2) Map `db_user` to break-glass accounts only. (3) Require pre-approved DDL ticket in correlation. (4) Retain statements hashed if raw SQL is sensitive.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Add-on for Microsoft Windows (742), Splunk DB Connect (if DB audit ingested).
• Ensure the following data sources are available: `index=db` `sourcetype="mssql:audit"` or `oracle:audit` — `object_name`, `statement`, `action`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Enable fine-grained audit on token vault schema. (2) Map `db_user` to break-glass accounts only. (3) Require pre-approved DDL ticket in correlation. (4) Retain statements hashed if raw SQL is sensitive.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=db sourcetype IN ("mssql:audit","oracle:audit") earliest=-7d
| where match(statement,"(?i)alter\\s+table|create\\s+index|hashbytes|truncate\\s+column|token")
| lookup pci_token_db_objects.csv object_name OUTPUT in_token_vault
| where in_token_vault="true"
| stats earliest(_time) as first_seen, latest(_time) as last_seen, values(statement) as stmts by object_name, db_user
| sort last_seen
```
