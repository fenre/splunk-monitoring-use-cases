# UC-22.1.49 · GDPR Storage Limitation Automation Evidence

## Description

Storage limitation requires defined retention and deletion. Beyond index retention (UC-22.1.4), this use case tracks automated purge jobs, API deletion endpoints, and backup rotation for personal-data datasets — proving deletion automation runs successfully.

## Value

Gives the responsible team auditor-ready evidence for GDPR compliance by automating regulatory compliance evidence using indexes app, backup and sourcetypes deletion_job. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Export dataset-level retention policy to `gdpr_retention_schedule.csv`; (2) instrument deletion jobs with structured logs; (3) alert data owners on failed or missed runs; (4) correlate with object storage lifecycle events; (5) include metrics in DPO quarterly report. Indexes required: `index=app`, `index=backup`; Sourcetypes: `sourcetype=deletion_job`.

## Detailed Implementation

Step 1 — Configure data collection
(1) Export dataset-level retention policy to `gdpr_retention_schedule.csv`; (2) instrument deletion jobs with structured logs; (3) alert data owners on failed or missed runs; (4) correlate with object storage lifecycle events; (5) include metrics in DPO quarterly report.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
| inputlookup gdpr_retention_schedule.csv
| join dataset_name [
    search index=app sourcetype="deletion_job" earliest=-7d
    | stats latest(job_status) as last_status latest(_time) as last_run by dataset_name
]
| eval overdue=if(last_status!="success" OR (now()-last_run)/86400 > scheduled_interval_days, 1, 0)
| where overdue=1
| table dataset_name, retention_days, scheduled_interval_days, last_status, last_run, overdue
```
