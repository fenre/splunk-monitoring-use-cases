# UC-22.11.17 · Cryptographic Erasure Verification After Decommission

## Description

Correlates disk wipe or crypto-shred completion events with asset retirement tickets so media and VMs that held account data cannot return to production without verified erasure evidence.

## Value

Gives the responsible team auditor-ready evidence for PCI DSS compliance by automating regulatory compliance evidence using indexes aws, itsm, sec and sourcetypes disk:wipe:log, snow:cmdb_ci. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Ensure wipe tooling emits `asset_id` matching CMDB. (2) Define `wipe_status` enumeration with `VERIFIED`. (3) Escalate open rows weekly. (4) Store closure export for PCI media controls. Indexes required: `index=aws`, `index=itsm`, `index=sec`; Sourcetypes: `sourcetype=disk:wipe:log`, `sourcetype=snow:cmdb_ci`.

## Detailed Implementation

• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Ensure wipe tooling emits `asset_id` matching CMDB. (2) Define `wipe_status` enumeration with `VERIFIED`. (3) Escalate open rows weekly. (4) Store closure export for PCI media controls.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=itsm sourcetype="snow:cmdb_ci" earliest=-30d
| where match(short_description,"(?i)decommission|retire") AND match(short_description,"(?i)CDE|PCI|payment")
| rename sys_id as asset_id, number as chg_ticket
| join type=left max=0 asset_id [
    search index=sec sourcetype="disk:wipe:log" earliest=-30d
    | stats max(wipe_status) as wipe_ok by asset_id
  ]
| where isnull(wipe_ok) OR wipe_ok!="VERIFIED"
| table chg_ticket, asset_id, short_description, wipe_ok
```
