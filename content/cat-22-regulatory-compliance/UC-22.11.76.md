# UC-22.11.76 · Penetration Test Finding Severity and Re-test Status

## Description

Tracks open penetration test findings and re-validation dates so critical issues from annual or incremental tests are remediated within agreed timelines and evidence is ready for PCI validation.

## Value

Gives the responsible team auditor-ready evidence for PCI DSS compliance by automating regulatory compliance evidence using indexes grc and sourcetypes grc:penetration_test. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Normalize severity strings. (2) Link `finding_id` to Jira epics. (3) Monthly steering committee view. (4) Store signed pentest report hash in GRC with Splunk snapshot. Indexes required: `index=grc`; Sourcetypes: `sourcetype=grc:penetration_test`.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=grc sourcetype="grc:penetration_test" earliest=-365d
| where match(engagement_scope,"(?i)CDE|payment|PCI)")
| where match(status,"(?i)open|accepted_risk")
| eval overdue_retest=if(isnotnull(retest_date) AND now()>retest_date,1,0)
| stats values(severity) as sev, max(overdue_retest) as overdue by finding_id, title
| where match(mvjoin(sev,","),"(?i)critical|high") OR overdue=1
| sort finding_id
```
