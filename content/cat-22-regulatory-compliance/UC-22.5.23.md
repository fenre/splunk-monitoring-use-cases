# UC-22.5.23 · MiFID II Best Execution — Client Limit Order Price Improvement vs Top of Book

## Description

Limit orders should receive fair treatment relative to displayed liquidity. Measuring frequency and magnitude of price improvement versus top-of-book at execution time supports conduct narratives for retail protection.

## Value

Limit orders should receive fair treatment relative to displayed liquidity. Measuring frequency and magnitude of price improvement versus top-of-book at execution time supports conduct narratives for retail protection.

## Implementation

(1) Validate `best_bid/ask` timestamps align with fill; (2) handle inverted books; (3) retail-only scope; (4) investigation workflow; (5) link to smart order router version.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: EMS and market data via HEC.
• Ensure the following data sources are available: `index=trading` `sourcetype="ems:fill"` (order_type, limit_px, best_bid, best_ask, exec_px, side, client_segment).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Validate `best_bid/ask` timestamps align with fill; (2) handle inverted books; (3) retail-only scope; (4) investigation workflow; (5) link to smart order router version.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=trading sourcetype="ems:fill" order_type="limit" client_segment="retail" earliest=-30d
| eval improvement_bps=if(side="B",(best_ask-exec_px)/best_ask*10000,(exec_px-best_bid)/best_bid*10000)
| where improvement_bps < 0
| stats count as worse_than_tob, median(improvement_bps) as med_imp by venue
| sort - worse_than_tob
```
