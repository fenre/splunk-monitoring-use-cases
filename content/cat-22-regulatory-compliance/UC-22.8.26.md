# UC-22.8.26 · SOC 2 CC9 — Change Authorization Dual-Control on Privileged Cloud Roles

## Description

CC9 addresses mitigation of fraud and management override. Ensuring IAM role attachments and policy updates have dual approval in PAM or ticketing prevents single-person override of segregation of duties—key for change authorization narratives.

## Value

Gives the responsible team auditor-ready evidence for SOC 2 compliance by automating regulatory compliance evidence using indexes aws, itsm and sourcetypes aws:cloudtrail, snow:change_request. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Replace join with robust ticket id extracted from CloudTrail `requestParameters` or session context; (2) integrate CyberArk session id where available; (3) emergency break-glass via lookup with post-incident review flag; (4) SOX-style quarterly sampling export; (5) align with CC6 logical access monitoring. Indexes required: `index=aws`, `index=itsm`; Sourcetypes: `sourcetype=aws:cloudtrail`, `sourcetype=snow:change_request`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Add-on for AWS (Splunkbase 1876), Splunk Add-on for ServiceNow (Splunkbase 1928).
• Ensure the following data sources are available: `index=aws` `sourcetype="aws:cloudtrail"` eventName IN ("AttachRolePolicy","PutRolePolicy") (requestParameters, userIdentity.arn, _time); `index=itsm` `sourcetype="snow:change_request"` with `u_dual_approval=true`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Replace join with robust ticket id extracted from CloudTrail `requestParameters` or session context; (2) integrate CyberArk session id where available; (3) emergency break-glass via lookup with post-incident review flag; (4) SOX-style quarterly sampling export; (5) align with CC6 logical access monitoring.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=aws sourcetype="aws:cloudtrail" eventName IN ("AttachRolePolicy","PutRolePolicy") earliest=-7d
| eval ticket=mvindex(split(requestParameters.policyArn,"/"),-1)
| join type=left userIdentity.arn [
    search index=itsm sourcetype="snow:change_request" u_dual_approval=true earliest=-14d
    | rename opened_by as approver1, u_second_approver as approver2
    | eval join_key=opened_by
    | table number, join_key
  ]
| where isnull(number)
| stats count by userIdentity.arn, eventName
| sort - count
```
