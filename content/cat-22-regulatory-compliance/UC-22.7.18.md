# UC-22.7.18 · Data Asset Classification — Sensitive Columns in Query Logs

## Description

Uses database audit logs tagged with sensitivity labels to prove data assets are known and classified for risk treatment prioritization.

## Value

Gives the responsible team auditor-ready evidence for NIST CSF compliance by automating regulatory compliance evidence using indexes database and sourcetypes db:audit. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Maintain sensitive column catalog; (2) exclude ETL service accounts; (3) quarterly access review input; (4) align to data governance council minutes. Indexes required: `index=database`; Sourcetypes: `sourcetype=db:audit`; Lookups: `sensitive_columns`, `sensitive_columns.csv`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk DB Connect (Splunkbase 2686).
• Ensure the following data sources are available: `db:audit` (table_name, column_name, user).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Maintain sensitive column catalog; (2) exclude ETL service accounts; (3) quarterly access review input; (4) align to data governance council minutes.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=database sourcetype=db:audit earliest=-24h
| lookup sensitive_columns.csv table_name column_name OUTPUT classification
| where classification="restricted"
| stats count by user, table_name
```
