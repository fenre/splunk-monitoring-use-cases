# UC-22.14.46 · Incident Monitoring — SOC Queue Depth and Severity Mix

## Description

Tracks open incident volume and severity for IR-5 incident monitoring (SOC operations).

## Value

Gives the responsible team auditor-ready evidence for NIST 800-53 compliance by automating regulatory compliance evidence. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## Implementation

(1) Ensure owners populated; (2) alert on `New` backlog thresholds; (3) weekly management review dashboard; (4) map to staffing model.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise Security (Splunkbase 263).
• Ensure the following data sources are available: `` `notable` `` macro fields.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Ensure owners populated; (2) alert on `New` backlog thresholds; (3) weekly management review dashboard; (4) map to staffing model.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
`notable` earliest=-7d status IN ("New","In Progress","Pending")
| stats count by urgency, status, owner
| sort - count
```

Understanding this SPL

**Incident Monitoring — SOC Queue Depth and Severity Mix (IR-5)** — Tracks open incident volume and severity for IR-5 incident monitoring (SOC operations).

Documented **Data sources**: `` `notable` `` macro fields. **App/TA** (typical add-on context): Splunk Enterprise Security (Splunkbase 263). The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

**Pipeline walkthrough**

• Invokes macro `notable` — in Search, use the UI or expand to inspect the underlying SPL.
• `stats` rolls up events into metrics; results are split **by urgency, status, owner** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.


Step 3 — Validate
Confirm that events are present in the index and that the search returns expected results. Compare with known good/bad scenarios if applicable. Verify field extractions and index permissions.

Step 4 — Operationalize
Add the search to a dashboard or set up alert actions (email, webhook, PagerDuty, etc.) as required. Document the use case in your runbook and assign an owner. Consider visualizations: Stacked bar (severity x status), Table (owner workload), Single value (open critical)
