# 22.15.39 — Secure Boot or Image Hash Mismatch on Industrial Appliance

**Criticality:** critical | **Difficulty:** advanced | **Status:** N/A

## Description

Correlates component attestation failures with subsequent process alarms to prioritize firmware integrity incidents.

## Value

Gives the responsible team auditor-ready evidence for IEC 62443 compliance by automating regulatory compliance evidence using indexes ot_edge, ot_hist and sourcetypes edge:attest, hist:alarm. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=ot_edge sourcetype="edge:attest" earliest=-7d
| where result!="OK" OR match(_raw,"(?i)hash mismatch|secure boot fail")
| join type=left device_id [ search index=ot_hist sourcetype="hist:alarm" earliest=-7d | stats count as alarms by device_id ]
| table _time, device_id, result, alarms
```

## Implementation

(1) Collect TPM/UEFI events from industrial gateways; (2) isolate device on mismatch; (3) compare to gold hash registry; (4) vendor RMA workflow. Indexes required: `index=ot_edge`, `index=ot_hist`; Sourcetypes: `sourcetype=edge:attest`, `sourcetype=hist:alarm`.

## Visualization

Table, Timeline, Single value (failed devices). Key fields: `]`, `alarms`, `device_id`, `result`.

## Compliance

- **IEC 62443** 2013-ongoing: CR 3.4 (partial) — Uplifted from contributing to partial: UC provides meaningful monitoring evidence for CR 3.4.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
