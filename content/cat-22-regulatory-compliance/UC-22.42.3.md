# UC-22.42.3 · Change rollback execution evidence — declared rollback vs actual

## Description

Continuously verifies that every change with a declared rollback plan has exercised that plan within policy cadence (default 180 days).

## Value

Turns rollback-plan existence from a documentation artefact into a tested capability.

## Implementation

(1) Export CR metadata to itsm_changes.csv; (2) have CI/CD emit rollback_test events; (3) run weekly; (4) on miss, schedule a rollback rehearsal.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform.
• Ensure the following data sources are available: ITSM CR records (ServiceNow), CI/CD pipeline audit (rollback_test events)..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Export CR metadata to itsm_changes.csv; (2) have CI/CD emit rollback_test events; (3) run weekly; (4) on miss, schedule a rollback rehearsal.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
| inputlookup itsm_changes.csv
| where rollback_plan_required="true"
| lookup rollback_tests.csv change_id OUTPUT last_tested
| eval age_days=round((now()-strptime(last_tested,"%Y-%m-%d"))/86400,0)
| where isnull(last_tested) OR age_days>180
| table change_id rollback_plan_required last_tested age_days
```

## Known False Positives

Emergency hotfix CRs may legitimately have a 'forward-fix-only' rollback strategy; annotate with rollback_type to exclude them.
