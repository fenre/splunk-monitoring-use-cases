# UC-22.48.5 · Vendor-master SoD — same identity creates vendor AND approves payment

## Description

Classic fraud-vector control: the ability to create a vendor and approve its payment cannot reside with a single identity.

## Value

Continuously enforces a control most organisations only sample.

## Implementation

(1) Ensure ERP emits both events; (2) run hourly; (3) auto-freeze suspicious vendor payments via AP workflow. Indexes required: `index=erp`; Sourcetypes: `sourcetype=*:ap_payment`, `sourcetype=oracle:vendor_master`, `sourcetype=sap:vendor_master`.

Step 1 — Configure data collection

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=erp sourcetype=sap:vendor_master OR sourcetype=oracle:vendor_master action=create earliest=-30d
| stats min(_time) as created_at values(created_by) as creators by vendor_id
| join vendor_id [search index=erp sourcetype=*:ap_payment action=approve earliest=-30d | stats values(approved_by) as approvers by vendor_id]
| eval overlap=mvfilter(match(creators,approvers))
| where mvcount(overlap)>0
| table vendor_id created_at creators approvers overlap
```
