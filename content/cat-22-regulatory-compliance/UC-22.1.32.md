# UC-22.1.32 · GDPR DPIA Completion Tracking Against High-Risk Processing

## Description

Where processing is likely to result in a high risk, the controller shall carry out a DPIA prior to processing. This use case reconciles the ROPA, system go-live dates, and DPIA register to flag processing that started or changed materially without a completed DPIA.

## Value

Where processing is likely to result in a high risk, the controller shall carry out a DPIA prior to processing. This use case reconciles the ROPA, system go-live dates, and DPIA register to flag processing that started or changed materially without a completed DPIA.

## Implementation

(1) Keep `high_risk` flags on ROPA entries per Art. 35(3); (2) join DPIA register by processing activity; (3) correlate major changes with DPIA refresh policy; (4) alert DPO on `MISSING` before production changes; (5) align with UC-22.1.14 for consolidated DPIA coverage views.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
| inputlookup gdpr_ropa_register.csv WHERE high_risk="true"
| lookup gdpr_dpia_register.csv processing_activity OUTPUT dpia_status, dpia_completed_date
| eval dpi_gap=case(
    dpia_status="completed", "OK",
    dpia_status="in_progress", "IN_PROGRESS",
    1=1, "MISSING")
| join processing_activity type=left [
    search index=itsm sourcetype="snow:change_request" earliest=-90d
    | where match(lower(short_description),"(?i)go.live|launch|migrate")
    | stats latest(start_date) as last_major_change by cmdb_ci
    | rename cmdb_ci as system_name
]
| where dpi_gap!="OK"
| table processing_activity, system_name, dpi_gap, dpia_status, last_major_change
```
