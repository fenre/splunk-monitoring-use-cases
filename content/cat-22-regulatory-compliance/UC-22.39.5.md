# UC-22.39.5 · Regulator-portal authentication failure during submission window

## Description

Joins IdP authentication failure events for a named set of regulator-portal service accounts against the currently open-incident register. An MFA failure in this window is a blocking event for notification SLA compliance.

## Value

Converts a class of silent failures (an expired portal API key, a stale MFA token) that has materialised in past incidents into an always-on monitoring pattern.

## Implementation

(1) Maintain a named macro regulator_service_accounts listing the IdP principals; (2) run every 5-10 min; (3) SOAR pages the DPO on hit.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform.
• Ensure the following data sources are available: IdP audit logs (okta:audit, azuread:audit), incident_register.csv with open_status flag..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Maintain a named macro regulator_service_accounts listing the IdP principals; (2) run every 5-10 min; (3) SOAR pages the DPO on hit.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=idp sourcetype=okta:audit OR sourcetype=azuread:audit earliest=-15m result=failure
| search user IN (`regulator_service_accounts`)
| join user [| inputlookup incident_register.csv | where status="open" | fields user=regulator_user incident_id]
| table _time user incident_id src_ip failure_reason
```
