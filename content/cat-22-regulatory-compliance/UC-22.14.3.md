# UC-22.14.3 · Audit Storage Capacity and Index Growth Guardrails

## Description

Detects indexes approaching disk or license limits that could cause loss of audit evidence, supporting AU-4 audit storage capacity management.

## Value

Detects indexes approaching disk or license limits that could cause loss of audit evidence, supporting AU-4 audit storage capacity management.

## Implementation

(1) Run from a scheduled search with `rest` rights; (2) compare `size_gb` to business thresholds by index class; (3) integrate license usage from `_internal` for combined risk scoring; (4) open change requests before forced rolling.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Enterprise / Splunk Cloud Platform (REST, DMC).
• Ensure the following data sources are available: `_internal` (metrics.log license), `/services/data/indexes` (index size, frozenTimePeriodInSecs).
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Run from a scheduled search with `rest` rights; (2) compare `size_gb` to business thresholds by index class; (3) integrate license usage from `_internal` for combined risk scoring; (4) open change requests before forced rolling.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
| rest /services/data/indexes splunk_server=local count=0
| search disabled=0 NOT title IN ("_*")
| eval size_gb=currentDBSizeMB/1024
| eval retention_days=round(frozenTimePeriodInSecs/86400,1)
| table title, size_gb, maxTotalDataSizeMB, retention_days, frozenTimePeriodInSecs
| sort - size_gb
```
