# UC-22.12.39 · SOX-ITGC ChangeMgmt.Approval — Segregation of duties in financial-system change approval

## Description

Detects (a) implementer = approver SoD violations and (b) approvals post-dating implementation.

## Value

Eliminates the two patterns that SOX auditors most frequently cite. The daily report is the control evidence.

## Implementation

(1) Maintain financial_systems_apps.csv; (2) capture ServiceNow workflow events; (3) schedule UC hourly; (4) violations open an immediate deficiency; (5) monthly CFO report. Indexes required: `index=change`; Sourcetypes: `sourcetype=servicenow:change`.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk Add-on for ServiceNow.
• Ensure the following data sources are available: `index=change` sourcetype=servicenow:change (workflow events)..
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
(1) Maintain financial_systems_apps.csv; (2) capture ServiceNow workflow events; (3) schedule UC hourly; (4) violations open an immediate deficiency; (5) monthly CFO report.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=change sourcetype=servicenow:change earliest=-7d
| stats latest(implementer) AS implementer latest(approver) AS approver latest(approval_date) AS approved_at latest(state_date) AS implemented_at BY change_id system
| lookup financial_systems_apps.csv system OUTPUT in_scope
| where in_scope="yes"
| eval sod_violation=if(implementer=approver,1,0)
| eval approval_before_implementation=if(approved_at<=implemented_at,1,0)
| where sod_violation=1 OR approval_before_implementation=0
| table change_id system implementer approver approved_at implemented_at sod_violation approval_before_implementation
```
