# 22.13.65 — Software Package Integrity — Signed Installer Verification Failures

**Criticality:** critical | **Difficulty:** advanced | **Status:** N/A

## Description

We capture endpoint or packaging pipeline events where installers fail signature or hash verification for software deployed to BES environments.

## Value

Gives the responsible team auditor-ready evidence for NERC CIP compliance by automating regulatory compliance evidence using indexes endpoint and sourcetypes ms:defender:behavior`, package:verify, package:verify`. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=endpoint sourcetype=package:verify bes_scope=1 earliest=-30d
| where match(result,"(?i)fail|invalid|unsigned")
| stats count by package_name, host, signer, result
| sort - count
```

## CIM SPL

```spl
| tstats summariesonly=t count from datamodel=Malware.Malware_Attacks by Malware_Attacks.dest | sort - count
```

## Implementation

(1) Instrument package repos to emit verification events. (2) Block deployment outside pipeline. (3) Page security on any failure. (4) Map to CIP-010 unauthorized software use case. Indexes required: `index=endpoint`; Sourcetypes: `sourcetype=ms:defender:behavior``, `sourcetype=package:verify`, `sourcetype=package:verify``.

## Visualization

Table (failures), Bar chart (by package), Single value (hosts)

## Compliance

- **NERC CIP** current: CIP-013-1 R2 Part 2.1 (partial) — Uplifted from contributing to partial: UC provides meaningful monitoring evidence for CIP-013-1 R2 Part 2.1.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
