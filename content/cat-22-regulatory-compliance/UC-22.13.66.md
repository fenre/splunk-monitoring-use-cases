# 22.13.66 — Vendor Security Incident Notification Receipt Tracking

**Criticality:** high | **Difficulty:** intermediate | **Status:** N/A

## Description

We log inbound vendor security bulletins and track acknowledgement SLAs for incidents that may affect BES software or services.

## Value

Gives the responsible team auditor-ready evidence for NERC CIP compliance by automating regulatory compliance evidence using indexes email and sourcetypes vendor:bulletin, vendor:bulletin`. Replaces manual evidence gathering with a continuous Splunk search that produces structured, time-stamped artefacts suitable for regulatory examination and internal audit review.

## SPL

```spl
index=email sourcetype=vendor:bulletin earliest=-90d severity IN ("high","critical")
| eval recv=strptime(received_at,"%Y-%m-%d %H:%M:%S")
| eval ack_epoch=strptime(acknowledged_at,"%Y-%m-%d %H:%M:%S")
| eval ack_h=round((ack_epoch-recv)/3600,2)
| where isnull(acknowledged_at) OR ack_h>72
| table bulletin_id, vendor, received_at, acknowledged_at, ack_h
```

## Implementation

(1) Parse vendor mailing lists into structured events. (2) Define acknowledgement workflow in ITSM. (3) Alert procurement and security. (4) Link to CIP-008 if operational impact. Indexes required: `index=email`; Sourcetypes: `sourcetype=vendor:bulletin`, `sourcetype=vendor:bulletin``.

## Visualization

Table (late acks), Single value (open critical), Timeline. Key fields: `ack_epoch`, `ack_h`, `acknowledged_at`, `bulletin_id`, `received_at`, `recv`.

## Compliance

- **NERC CIP** current: CIP-013-1 R3 Part 3.1 (partial) — Uplifted from contributing to partial: UC provides meaningful monitoring evidence for CIP-013-1 R3 Part 3.1.

## Known False Positives

Administrative tasks, scheduled jobs or platform updates can match this pattern — correlate with change management, maintenance windows and user role before raising severity.
