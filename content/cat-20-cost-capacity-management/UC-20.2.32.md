# UC-20.2.32 · GPU Pool Utilization for ML Workload Capacity

## Description

GPU nodes are expensive; low average utilization wastes capital while queue spikes delay training service levels. Tracking streaming-multiprocessor utilization and peak load by host informs autoscaler bounds and purchase versus spot decisions.

## Value

GPU nodes are expensive; low average utilization wastes capital while queue spikes delay training service levels. Tracking streaming-multiprocessor utilization and peak load by host informs autoscaler bounds and purchase versus spot decisions.

## Implementation

(1) Deploy DCGM on GPU nodes and normalize `gpu_index`. (2) Tag hosts with workload type such as training versus inference. (3) Right-size node pools when underused persists fourteen days; scale out when fleet_p95 exceeds ninety-two.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=ml_infra sourcetype="dcgm:gpu" earliest=-7d
| eval util=tonumber(coalesce(gpu_sm_utilization, sm_util_pct))
| bin _time span=1h
| stats avg(util) as avg_sm perc95(util) as p95_sm by _time, host, gpu_index
| stats avg(avg_sm) as fleet_avg max(p95_sm) as fleet_p95 by host
| eval underused=if(fleet_avg<35 AND fleet_p95<70,1,0)
| where underused=1 OR fleet_p95>92
| table host, fleet_avg, fleet_p95
```
