# UC-13.1.35 · User Search Activity Audit (_audit)

## Description

Auditing who ran which searches supports insider-threat investigations and segregation of duties.

## Value

Auditing who ran which searches supports insider-threat investigations and segregation of duties.

## Implementation

Retain per policy. Report on after-hours or high-volume search users. Exclude known service accounts via lookup.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Audit trail (built-in).
• Ensure the following data sources are available: `index=_audit` `action=search`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Retain per policy. Report on after-hours or high-volume search users. Exclude known service accounts via lookup.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=_audit action=search info=started
| stats count, dc(search) as distinct_searches by user, app
| sort -count
```

Understanding this SPL

**User Search Activity Audit (_audit)** — Auditing who ran which searches supports insider-threat investigations and segregation of duties.

Documented **Data sources**: `index=_audit` `action=search`. **App/TA** (typical add-on context): Audit trail (built-in). The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: _audit.

**Pipeline walkthrough**

• Scopes the data: index=_audit. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `stats` rolls up events into metrics; results are split **by user, app** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Orders rows with `sort` — combine with `head`/`tail` for top-N patterns.


Step 3 — Validate
Confirm that events are present in the index and that the search returns expected results. Compare with known good/bad scenarios if applicable. Verify field extractions and index permissions.

Step 4 — Operationalize
Add the search to a dashboard or set up alert actions (email, webhook, PagerDuty, etc.) as required. Document the use case in your runbook and assign an owner. Consider visualizations: Table (users by search volume), Heatmap (hour × user).
