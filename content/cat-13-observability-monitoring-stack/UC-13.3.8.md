# UC-13.3.8 · Log Ingestion Backlog and Lag

## Description

Ingestion backlog or lag causes delayed alerting and search. Monitoring lag per source ensures data freshness and pipeline health.

## Value

Ingestion backlog or lag causes delayed alerting and search. Monitoring lag per source ensures data freshness and pipeline health.

## Implementation

Monitor forwarder and indexer queue metrics. Alert when queue depth or event lag (now - event time) exceeds threshold. Report on lag by source and index.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk _internal, forwarder metrics.
• Ensure the following data sources are available: Indexer acknowledgment, forwarder queue depth, event timestamps.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Monitor forwarder and indexer queue metrics. Alert when queue depth or event lag (now - event time) exceeds threshold. Report on lag by source and index.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=_internal source=*metrics* group=queue
| stats latest(current_size) as queue_depth by host, name
| where queue_depth > 1000
| table host, name, queue_depth
```

Understanding this SPL

**Log Ingestion Backlog and Lag** — Ingestion backlog or lag causes delayed alerting and search. Monitoring lag per source ensures data freshness and pipeline health.

Documented **Data sources**: Indexer acknowledgment, forwarder queue depth, event timestamps. **App/TA** (typical add-on context): Splunk _internal, forwarder metrics. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: _internal.

**Pipeline walkthrough**

• Scopes the data: index=_internal. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `stats` rolls up events into metrics; results are split **by host, name** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Filters the current rows with `where queue_depth > 1000` — typically the threshold or rule expression for this monitoring goal.
• Pipeline stage (see **Log Ingestion Backlog and Lag**): table host, name, queue_depth


Step 3 — Validate
Confirm that events are present in the index and that the search returns expected results. Compare with known good/bad scenarios if applicable. Verify field extractions and index permissions.

Step 4 — Operationalize
Add the search to a dashboard or set up alert actions (email, webhook, PagerDuty, etc.) as required. Document the use case in your runbook and assign an owner. Consider visualizations: Table (hosts with backlog), Single value (max lag minutes), Line chart (queue depth trend).
