# UC-13.2.11 · KPI Threshold Violation Trending

## Description

Trending KPI breaches over time shows chronic vs transient service issues and validates threshold tuning.

## Value

Trending KPI breaches over time shows chronic vs transient service issues and validates threshold tuning.

## Implementation

Baseline breach counts per KPI with `streamstats` or `predict`. Alert on sustained elevation vs one-off spikes. Feed results into Episode Review for service owners.

## Detailed Implementation

Prerequisites
• Install and configure the required add-on or app: Splunk ITSI.
• Ensure the following data sources are available: `index=itsi_summary`, `itsi_notable:audit`.
• For app installation, inputs.conf, and Splunk directory layout, see the Implementation guide: docs/implementation-guide.md

Step 1 — Configure data collection
Baseline breach counts per KPI with `streamstats` or `predict`. Alert on sustained elevation vs one-off spikes. Feed results into Episode Review for service owners.

Step 2 — Create the search and alert
Run the following SPL in Search (then save as report or alert; adjust time range and threshold as needed):

```spl
index=itsi_summary severity_value>=3
| timechart span=1h count by service_name, kpi_name
| streamstats window=24 avg(count) as baseline by kpi_name
| where count > baseline * 2
```

Understanding this SPL

**KPI Threshold Violation Trending** — Trending KPI breaches over time shows chronic vs transient service issues and validates threshold tuning.

Documented **Data sources**: `index=itsi_summary`, `itsi_notable:audit`. **App/TA** (typical add-on context): Splunk ITSI. The SPL below should target the same indexes and sourcetypes you configured for that feed—rename `index=` / `sourcetype=` if your deployment differs.

The first pipeline stage scopes events using **index**: itsi_summary.

**Pipeline walkthrough**

• Scopes the data: index=itsi_summary. Cross-check against **Data sources** above so indexes and sourcetypes match your ingestion.
• `timechart` plots the metric over time using **span=1h** buckets with a separate series **by service_name, kpi_name** — ideal for trending and alerting on this use case.
• `streamstats` rolls up events into metrics; results are split **by kpi_name** so each row reflects one combination of those dimensions (useful for per-host, per-user, or per-entity comparisons for this use case).
• Filters the current rows with `where count > baseline * 2` — typically the threshold or rule expression for this monitoring goal.

Enable Data Model Acceleration (and metric indexes for `mstats`) for the models or datasets referenced above; otherwise `tstats`/`mstats` may return no results from summaries.


Step 3 — Validate
Confirm that events are present in the index and that the search returns expected results. Compare with known good/bad scenarios if applicable. Verify field extractions and index permissions.

Step 4 — Operationalize
Add the search to a dashboard or set up alert actions (email, webhook, PagerDuty, etc.) as required. Document the use case in your runbook and assign an owner. Consider visualizations: Line chart (breaches per KPI), Heatmap (service × hour), Table (KPIs above baseline).
