# Per-UC Splunk dashboard scaffolds

This repository ships a **deterministic generator** that emits Splunk dashboard
scaffolds for every use case in the catalogue. Each UC produces two files under
`dist/dashboards/UC-X.Y.Z/`:

| File | Format | Purpose |
|------|--------|---------|
| `simple.xml` | Classic Simple XML (`<dashboard version="1.1">`) | Drop into `<app>/default/data/ui/views/` |
| `studio.xml` | Dashboard Studio (`<dashboard version="2">` + JSON CDATA) | Same path; Splunk 10.x / Cloud Dashboard Studio |

Generated output is **gitignored** (`dist/`). CI regenerates it on every PR.

## Quick start

```bash
# Generate all dashboard pairs (~7,900 UCs)
make generate-dashboards

# Validate structure + SPL anti-patterns + determinism
make audit-dashboards

# Single UC
PYTHONPATH=src python3 -m splunk_uc generate-dashboards --only UC-1.1.1
```

## Import path

1. Build or copy the pair for your UC from `dist/dashboards/UC-X.Y.Z/`.
2. Place the XML file in your Splunk app:

   ```
   <APP_ID_PLACEHOLDER>/default/data/ui/views/UC-1.1.1-simple.xml
   <APP_ID_PLACEHOLDER>/default/data/ui/views/UC-1.1.1-studio.xml
   ```

3. Replace `<APP_ID_PLACEHOLDER>` with your app folder name (must match
   `[package] id` in `app.conf`).

4. Reload the app or restart Splunk. Open **Dashboards** in Splunk Web.

## Placeholder replacement

Every panel queries **real indexes and sourcetypes** taken from the UC sidecar
SPL. Before go-live, search each dashboard XML for placeholders and substitute
your environment values:

| Token | Replace with |
|-------|----------------|
| `<INDEX_PLACEHOLDER>` | Your index name (e.g. `os`, `network`) |
| `<SOURCETYPE_PLACEHOLDER>` | Your sourcetype (e.g. `cpu`, `cisco:sdwan:event`) |
| `<APP_ID_PLACEHOLDER>` | Your Splunk app id |

If the UC already specifies `index=` / `sourcetype=` in its SPL, those values
are copied verbatim — you only need to confirm they match your deployment.

## Panel layout

Each scaffold includes three panels:

1. **Hit count** — single value (`stats count` or the UC's terminating aggregation)
2. **Event trend** — line chart (`timechart` when present in the UC SPL, otherwise `timechart span=1h count`)
3. **Sample results** — table (`head 100` + `table *`)

To add panels, edit the template under `templates/dashboards/` and regenerate.
Do **not** hand-edit files under `dist/dashboards/` — they are overwritten on
every generator run.

## Why we never use `| makeresults`

The legacy `scripts/generate_catalog_dashboard.py` demo portfolio used
`| makeresults` to fabricate chart data. That pattern:

- Bypasses your real ingestion pipeline
- Shows non-zero values when indexes are empty
- Fails Splunk Cloud / VISTA validation gates

The per-UC generator **always** starts from `index=` + `sourcetype=` (from the
UC SPL). When no data exists, panels correctly show zero or empty — which is
what operators need before trusting a dashboard.

Other banned patterns (enforced by `audit-dashboards --check`):

- `random()` for jittery demo values
- `coalesce(field, 87.3)`-style hardcoded numeric fallbacks
- `viz.*` Dashboard Studio types (use `splunk.*` instead)
- Multi-pipe one-liners (every `|` must start its own line)

## Validate after editing

```bash
# Regenerate + audit (CI runs the same chain)
make generate-dashboards
make audit-dashboards

# Or directly:
PYTHONPATH=src python3 -m splunk_uc audit-dashboards --check
```

The audit gate checks:

- XML well-formedness (Simple XML + Studio wrapper)
- JSON parseability inside Studio `<definition><![CDATA[ … ]]>`
- SPL anti-patterns listed above
- Pipe-per-line formatting
- XML entity escaping in `<label>` / `<description>` (`&` → `&amp;`)
