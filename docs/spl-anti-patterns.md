# SPL anti-pattern audit

The `audit-spl-anti-patterns` verb scans every UC sidecar under
`content/cat-*/UC-*.json` for documented SPL **style and risk**
anti-patterns. It complements the existing SPL audits:

| Audit | What it catches |
| --- | --- |
| `audit-spl-grammar` | Structural bugs (invalid `stats span=`, leading pipe, …) |
| `audit-spl-hallucinations` | Unknown commands, bad CIM datasets |
| `audit-spl-references` | Fabricated macros, sourcetypes, datamodel paths |
| **`audit-spl-anti-patterns`** | **Quality anti-patterns** (join, makeresults, random(), …) |

## Queue concept (Lane N)

The audit does **not** auto-fix sidecars. It emits a maintainer queue:

- `dist/audits/spl-anti-patterns.json` — machine-readable match list
- `dist/audits/spl-anti-patterns.md` — human summary (top offenders, histograms)

Lane N maintainers pick UCs from the queue and fix them by hand. After
each burndown tranche, tighten the CI ratchet (see below).

## Quick start

```bash
# Full report (writes dist/audits/*)
PYTHONPATH=src python3 -m splunk_uc audit-spl-anti-patterns

# CI-style gate (high severity only)
PYTHONPATH=src python3 -m splunk_uc audit-spl-anti-patterns --check --severity high

# Investigate one pattern
PYTHONPATH=src python3 -m splunk_uc audit-spl-anti-patterns \
  --include-pattern ANTIPAT-001 --out /tmp/audits
```

Makefile targets: `make audit-spl-anti-patterns` (report),
`make audit-spl-anti-patterns-check` (CI gate).

## Pattern catalogue

Patterns live in [`data/spl-anti-patterns.json`](../data/spl-anti-patterns.json).
Each entry carries:

| Field | Purpose |
| --- | --- |
| `id` | Stable id (`ANTIPAT-001`, …) |
| `pattern` / `pattern_kind` | `regex` today; `semantic` and `ast` reserved |
| `severity` | `low` / `medium` / `high` |
| `category` | Grouping (`performance`, `fabrication`, …) |
| `remediation` | One-line fix guidance |
| `doc_anchor` | Link into `.cursor/rules/splunk-conf-and-spl.mdc` |

### Adding a new pattern

1. Add an entry to `data/spl-anti-patterns.json` (keep deterministic key order:
   `id`, `name`, `pattern`, `pattern_kind`, `severity`, `category`,
   `description`, `remediation`, `doc_anchor`).
2. Extend `tests/splunk_uc/audits/test_spl_anti_patterns.py` with a positive
   snippet that fires the rule.
3. Run `pytest tests/splunk_uc/audits/test_spl_anti_patterns.py`.
4. Re-run the audit against `content/` and update the CI `--limit` if the
   new rule surfaces existing catalogue debt you accept temporarily.

For `pattern_kind: semantic`, implement the detector in
`_semantic_matches()` inside `spl_anti_patterns.py` keyed on `entry.id`.

## Severity rubric

| Severity | Meaning | Examples |
| --- | --- | --- |
| **high** | Breaks performance, fabricates data, or hides empty results | `join`, `makeresults`, `random()`, `| search` after `| stats`, unbounded `transaction`, `earliest=0` |
| **medium** | Review burden or moderate runtime risk | hardcoded `coalesce` decimals, leading wildcards, unbounded `streamstats`, pipe-per-line violations |
| **low** | Authoring discipline nits | missing `index=` / `sourcetype=` when the search shape expects them |

## False-positive expectations

Some rules are intentionally conservative:

- **`ANTIPAT-008` / `ANTIPAT-009`** — skipped when the SPL uses `tstats
  from datamodel=`, `inputlookup`, or other generating commands.
- **`ANTIPAT-010`** — flags physical lines with more than one unquoted
  `|`; SPL inside markdown prose is not scanned (only `` ```spl `` fences
  in `detailedImplementation`).
- **`ANTIPAT-005`** — flags real `| search` after `| stats`; this is
  usually a genuine performance bug, not a false positive.

Tune severity before adding exclusions; prefer `--exclude-pattern` in
investigation, not permanent catalogue edits.

## CI ratchet

CI runs:

```yaml
python3 -m splunk_uc audit-spl-anti-patterns --check --severity high --limit 2000
```

The `--limit` allows the current high-severity backlog while **blocking
new** high-severity offenders. As Lane N drains the queue:

1. Lower `--limit` in `.github/workflows/validate.yml` to the new count.
2. Optionally raise `--severity` to `medium` once high-severity debt is gone.

Unit tests use a hermetic fixture under `tests/splunk_uc/audits/` and
assert that a synthetic offender fails `--check` at `--limit 0`.

## Scanned fields

Per UC sidecar:

- `spl`
- `qs` (reserved alias; not populated today)
- `detailedImplementation.spl` when nested, otherwise `` ```spl `` code
  fences inside the `detailedImplementation` string

Non-SPL narrative fields are ignored.
