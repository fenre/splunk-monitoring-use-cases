## The big idea

Until now, the catalog had 6,304 use cases with SPL queries, references, and metadata — but **no way to know which ones actually work, where each one came from, or which categories are solid vs. thin**.

v6.0 adds five new systems that answer those questions automatically, on every build, with zero manual upkeep.

---

## What's new, in one sentence each

| System | What it does | File you'd look at |
|---|---|---|
| **Sample events** | Real log snippets you can feed into Splunk to prove a detection fires | `samples/UC-<id>/` |
| **Test harness** | Runs each UC's SPL against its sample logs in a real Splunk container | `scripts/run_uc_tests.py` |
| **Cloud compatibility audit** | Flags SPL that would fail Splunk Cloud's AppInspect before you ship | `docs/splunk-cloud-compat.md` |
| **Provenance ledger** | For every UC, records where the detection idea came from (Splunk docs? MITRE? a blog?) | `docs/provenance-coverage.md` |
| **Quality scorecard** | Grades every category Gold / Silver / Bronze / Needs work on six signals | `docs/scorecard.md` |

## 1. Sample event fixtures

**What it is:** A new `samples/` folder with real log lines for 15 top use cases — Linux auth failures, Windows Sysmon process creation, Palo Alto traffic logs, AWS CloudTrail, Kubernetes audit, etc.

**Why it matters:** Without real data, no one can prove an SPL query works. Now every UC with a sample can be automatically tested end-to-end.

**Structure:**

```
samples/
└── UC-<id>/
    ├── manifest.yaml     # JSON-Schema-validated metadata
    ├── positive.log      # events that should trigger the detection
    └── negative.log      # (optional) events that should NOT trigger
```

**Next step (v6.1):** expand from 15 → 200 fixtures.

---

## 2. UC test harness

**What it is:** A script that takes each use case, feeds its sample logs into a disposable Splunk Docker container, runs the SPL query, and checks the results match expectations.

**How it's wired in:**
- **On every PR:** quick validation (no container needed) — runs in ~30 seconds
- **On pushes to main:** full end-to-end test in a real Splunk 9.4 container
- **Output:** JUnit XML that GitHub Actions renders as a pass/fail summary

**Why it matters:** Breaking changes to SPL get caught automatically, not after a user reports "this doesn't work in my environment."

---

## 3. Splunk Cloud compatibility audit

**What it is:** An automated scanner that checks every SPL query and every packaged `.conf` file for things that would be rejected by Splunk Cloud or AppInspect — custom search commands, scripted inputs, `| runshellscript`, unbounded `| map`, Python 2 directives, etc.

**First audit results:**
- **0 pack-level findings** — all Splunk apps in the repo are Cloud-clean
- **5 SPL warnings** — all legitimate `dbxquery` callouts (expected; DB Connect isn't fully Cloud-supported)

**How it's wired in:** Runs on every PR via `validate.yml`. Build fails if any new `severity=fail` pattern slips in.

## 4. Provenance ledger

**What it is:** For each of the 6,304 use cases, the system looks at every URL in the `References:` section and classifies the citation into one of 9 categories:

| Category | Authority | % of UCs |
|---|---|---|
| Splunk official docs | 1.00 | **72 %** |
| Vendor official docs (Cisco, MSFT, etc.) | 1.00 | **9 %** |
| Threat intelligence (Mandiant, Unit 42, CISA…) | 0.80 | **7 %** |
| MITRE ATT&CK | 1.00 | **5 %** |
| NIST / CIS / ISO / PCI | 1.00 | **1.5 %** |
| Splunk blog | 0.60 | 0.7 % |
| Community blogs | 0.50 | 2 % |
| Unclassified | 0.30 | 2.4 % |
| Contributor (no URL) | 0.20 | <0.1 % |

**Dashboard integration:** Each UC card now shows a **colour-coded source badge** with a hover tooltip. Users can instantly see "is this from official Splunk docs, or is it a community blog?"

**API:** The full ledger is available at `GET /provenance.json` for anyone building their own tooling.

## 5. Quality scorecard

**What it is:** A single page that grades every category on six quality signals, rolled into one composite score:

| Dimension | Weight | Why |
|---|---|---|
| References % | 25 % | Un-cited detections are unverifiable |
| Provenance authority | 25 % | Cited from official docs beats cited from a random blog |
| Freshness (last-reviewed) | 15 % | Old detections go stale |
| Known false positives % | 15 % | No KFP = fragile in production |
| MITRE ATT&CK coverage | 10 % | Security UCs should map to techniques |
| Sample fixtures % | 10 % | Proof it runs |

### First grading snapshot

| Grade | Meaning | Categories | UCs |
|---|---|---|---|
| Gold (≥85) | Production-ready | 0 | 0 |
| **Silver (70–84)** | Solid with minor gaps | **3** | **2,611** |
| **Bronze (55–69)** | Usable, needs attention | **4** | **1,899** |
| Needs work (<55) | Authoring effort required | 16 | 1,794 |

**Top performers:**
- Silver — Network Security & Zero Trust — 74.2
- Silver — Identity & Access Management — 73.5
- Silver — Security Infrastructure — 70.5

**Weakest area:** Most "Needs work" categories have great citations already but lack KFP guidance, reviewed-dates, and sample fixtures — so one coordinated pass could push many of them into Bronze/Silver.

## How it all stays in sync

Every time `build.py` runs:

```
1. Build catalog.json, data.js, api/*.json, llms.txt, sitemap.xml   (existing)
2. → Regenerate provenance.json + provenance.js + coverage report   (NEW)
3. → Regenerate scorecard.json + scorecard.md                       (NEW)
```

The CI pipeline (`.github/workflows/validate.yml`) refuses to merge any PR where these generated files are out of sync, so they can never drift.

## What a user sees

- **On the dashboard:** every UC card now shows a source badge ("Splunk Official" in blue, "MITRE ATT&CK" in red, "Community" in grey, etc.) next to the existing status/freshness chips.
