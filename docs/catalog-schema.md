# catalog.json ‚Äî Schema and Usage

The **machine-readable catalog** of use cases is generated by `build.py` and written as **`catalog.json`** in the repository root. It contains the same data as `data.js` (used by the dashboard), in JSON form, for scripts, tools, and integrations.

---

## Obtaining the catalog

- **Local:** Run `python3 build.py` in the repo root. Output: `catalog.json`.
- **Hosted:** If the repo is deployed to GitHub Pages, you can reference the raw file, for example:
  `https://<owner>.github.io/<repo>/catalog.json` (if you add `catalog.json` to the Pages build) or the raw GitHub URL for the default branch:  
  `https://raw.githubusercontent.com/<owner>/<repo>/main/catalog.json`

---

## Top-level keys

| Key | Type | Description |
|-----|------|-------------|
| `DATA` | array | All categories and use cases (see [DATA structure](#data-structure) below). |
| `CAT_META` | object | Category metadata: icon, description, quick tip (see [CAT_META](#cat_meta)). |
| `CAT_GROUPS` | object | Group names to category IDs for filtering (see [CAT_GROUPS](#cat_groups)). |
| `EQUIPMENT` | array | Equipment/TA definitions for ‚Äúfilter by technology‚Äù (see [EQUIPMENT](#equipment)). |
| `implementationRoadmap` | object | Optional, v1.4+. Precomputed crawl/walk/run rollout plan per category (see [implementationRoadmap](#implementationroadmap) below). Emitted only when at least one UC declares a `wave`. |

---

## DATA structure

`DATA` is an array of **category** objects, one per file `cat-NN-*.md`. Each category has:

| Field | Type | Description |
|-------|------|-------------|
| `i` | number | Category ID (1‚Äì23). |
| `n` | string | Category name (e.g. `"Server & Compute"`). |
| `src` | string | Source markdown filename under `use-cases/` (e.g. `"cat-01-server-compute.md"`). Set by `build.py` for links to the repo. |
| `s` | array | Subcategories (see below). |

Each element of `s` is a **subcategory**:

| Field | Type | Description |
|-------|------|-------------|
| `i` | string | Subcategory ID (e.g. `"1.1"`). |
| `n` | string | Subcategory name (e.g. `"Linux Servers"`). |
| `u` | array | Use cases (see below). |

Each element of `u` is a **use case**. The **UC-ID** is the use case‚Äôs `i` and matches the pattern `category.subcategory.index` (e.g. `1.1.1` = first use case under ‚Äú1.1 Linux Servers‚Äù). Optional fields may be omitted or empty strings/arrays.

### Use case fields (abbreviations and meaning)

| Key | Type | Description |
|-----|------|-------------|
| `i` | string | Use case ID (e.g. `"1.1.1"`). Full UC-ID: `UC-{i}`. |
| `n` | string | Title / name. |
| `c` | string | **Criticality:** `critical` \| `high` \| `medium` \| `low`. |
| `f` | string | **Difficulty:** `beginner` \| `intermediate` \| `advanced` \| `expert`. |
| `v` | string | **Value:** why the use case matters. |
| `t` | string | **App/TA:** Splunk add-on(s) or app(s). |
| `d` | string | **Data sources:** log/metric sources. |
| `q` | string | **SPL:** main search query. |
| `qs` | string | **CIM SPL:** tstats/accelerated query (optional). |
| `m` | string | **Implementation:** short deployment summary. |
| `md` | string | **Detailed implementation:** step-by-step (optional). When not authored in markdown, `build.py` generates this text; it includes **Understanding this SPL** with a short intro tied to the use case (**Value**, **Data sources**, **App/TA**), a **Pipeline walkthrough** with per-command detail, and optional **Understanding this CIM / accelerated SPL** when `qs` is present. |
| `z` | string | **Visualization:** suggested panels/charts. |
| `a` | array | **CIM models:** e.g. `["Performance", "Network_Traffic"]` (optional). |
| `dma` | string | **Data model acceleration** note (optional). |
| `schema` | string | **Schema:** e.g. `CIM`, `OCSF`, or `OCSF: authentication` (optional). |
| `mtype` | array | **Monitoring type:** e.g. `["Performance", "Capacity"]` (optional). |
| `kfp` | string | **Known false positives** (optional). |
| `refs` | string | **References:** URLs (optional). |
| `mitre` | array | **MITRE ATT&CK** technique IDs (optional). |
| `dtype` | string | **Detection type:** e.g. TTP, Anomaly, Baseline (optional). |
| `sdomain` | string | **Security domain** (optional). |
| `reqf` | string | **Required fields** for the search (optional). |
| `script` | string | **Script example** for scripted inputs (optional). |
| `premium` | string | **Premium apps** (ES, ITSI, SOAR) when required (optional). |
| `hw` | string | **Equipment models:** specific hardware models (optional). |
| `e` | array | **Equipment IDs:** from `EQUIPMENT` (matched by TA string). |
| `em` | array | **Equipment model IDs** (optional). |
| `status` | string | **Quality status:** `verified` \| `community` \| `draft` (optional, v5.1+). |
| `reviewed` | string | **Last reviewed:** ISO date `YYYY-MM-DD` (optional, v5.1+). |
| `sver` | string | **Splunk versions:** e.g. `9.2+`, `9.3+`, `Cloud` (optional, v5.1+). |
| `rby` | string | **Reviewer:** GitHub handle or `N/A` (optional, v5.1+). |
| `wv` | string | **Wave:** `crawl` \| `walk` \| `run` (optional, v1.4+). Groups UCs into implementation phases; see [Implementation ordering](use-case-fields.md#implementation-ordering-optional-v14) for authoring guidance. |
| `pre` | array | **Prerequisite UCs:** upstream UC IDs in `UC-X.Y.Z` form, e.g. `["UC-1.1.1", "UC-13.1.1"]` (optional, v1.4+). Deduplicated and sorted by `build.py`. |

## CAT_META

Maps category ID (as string) to metadata from `use-cases/INDEX.md`:

```json
{
  "1": {
    "icon": "monitor",
    "desc": "Linux, Windows, macOS endpoint and server monitoring...",
    "quick": "Deploy Splunk_TA_nix or Splunk_TA_windows..."
  },
  "2": { ... }
}
```

| Field | Description |
|-------|-------------|
| `icon` | Icon identifier (e.g. for UI). |
| `desc` | Category description. |
| `quick` | Quick tip (optional). |

---

## CAT_GROUPS

Maps group names to lists of category IDs (numbers). Used by the dashboard filter chips and for ‚Äúfilter by domain‚Äù:

```json
{
  "infra":      [1, 2, 5, 6, 15, 18, 19],
  "security":   [9, 10, 17],
  "cloud":      [3, 4, 20],
  "app":        [7, 8, 11, 12, 13, 14, 16],
  "industry":   [21],
  "compliance": [22],
  "business":   [23]
}
```

## implementationRoadmap

Optional, v1.4+. Computed by `build.py` from every UC's `wv` + `pre` fields. The structure is `{ category_id (string) ‚Üí { wave_key ‚Üí [uc_id_full, ...] } }`. Only categories that have at least one UC with a wave appear; each wave bucket is sorted by `(subcategory, uc_id)` so consumers can render a stable rollout plan without re-sorting.

```json
{
  "implementationRoadmap": {
    "1": {
      "crawl":      ["UC-1.1.1", "UC-1.1.4"],
      "walk":       ["UC-1.1.2"],
      "run":        ["UC-1.2.5"],
      "unassigned": ["UC-1.1.3", "UC-1.1.5"]
    }
  }
}
```

| Wave key | Meaning |
|----------|---------|
| `crawl` | Foundation ‚Äî platform, TAs, and primary data sources. Implement first. |
| `walk` | Intermediate ‚Äî extends or correlates foundation data. |
| `run` | Advanced ‚Äî cross-source correlation, ML, multi-UC orchestration. |
| `unassigned` | UC has not been classified yet. Emitted so consumers can surface coverage gaps. |

`build.py` also exposes the same structure in the browser as `window.ROADMAP` (served from `data.js`), which the dashboard reads to render the per-category "Crawl ‚Üí Walk ‚Üí Run" view.

## EQUIPMENT

Array of equipment/technology entries. Each entry has:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Slug (e.g. `"linux"`, `"cisco"`). |
| `label` | string | Display label. |
| `tas` | array | Substrings to match in use case App/TA (`t`). |
| `models` | array | Optional. Finer-grained models: `{ "id", "label", "tas" }`. |

Use `e` and `em` on a use case to know which equipment IDs (and model IDs) it matches.

## Example: load and use in a script

**List all use case IDs and titles:**

```python
import json
with open("catalog.json", "r", encoding="utf-8") as f:
    catalog = json.load(f)

for cat in catalog["DATA"]:
    for sub in cat["s"]:
        for uc in sub["u"]:
            uc_id = f"UC-{uc['i']}"
            print(uc_id, uc["n"])
```

**Get SPL for a specific UC-ID (e.g. 1.1.1):**

```python
target = "1.1.1"
for cat in catalog["DATA"]:
    for sub in cat["s"]:
        for uc in sub["u"]:
            if uc["i"] == target:
                print(uc.get("q", ""))
                break
```

**Filter use cases by criticality:**

```python
critical = [
    (f"UC-{uc['i']}", uc["n"])
    for cat in catalog["DATA"]
    for sub in cat["s"]
    for uc in sub["u"]
    if uc.get("c") == "critical"
]
```

**Resolve category name from CAT_META:**

```python
cat_id = "1"
meta = catalog["CAT_META"].get(cat_id, {})
print(meta.get("desc", ""))
```

**List categories in a group (e.g. infra):**

```python
group = catalog["CAT_GROUPS"]["infra"]
cat_names = [
    cat["n"] for cat in catalog["DATA"]
    if cat["i"] in group
]
```

## Versioning and build

- `catalog.json` is generated; do not edit it by hand.
- Re-run `python3 build.py` after changing `use-cases/*.md` or `use-cases/INDEX.md`.
- The schema may gain new optional use case fields as the use case markdown format is extended; existing keys remain backward compatible.
