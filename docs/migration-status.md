- [x] P8 — Metrics history snapshots + per-release trend record (v9.x, 2026-05-09): closes the "trend snapshots committed only at release tags" subtask of repo-overhaul plan §P8. Step 1 (the build-time emitter, above) ships a fresh `dist/metrics.json` on every build, but that file is ephemeral — the next build overwrites it. Step 2 captures one **verbatim** copy of `dist/metrics.json` per release into `data/metrics-history/<VERSION>.json`, plus a sorted `data/metrics-history/index.json` listing every release ever captured. The verbatim-copy choice is deliberate: no synthetic fields are injected at write-time, so the snapshot validates against the same `schemas/v2/metrics.schema.json` (which has `additionalProperties: false`) as the live build artefact and downstream tooling can consume snapshots with the same parser. - [x] P8 — Build telemetry per stage (v9.x, 2026-05-09): closes the "build telemetry per stage" subtask of repo-overhaul plan §P8. Captures wall-clock duration for every pipeline stage so pre-merge benchmarks can spot regressions before they ship. `tools/build/render_telemetry.py` consumes a list of `(stage, duration_ms)` records collected by `tools/build/build.py:_run_once` (each `_stage_*` invocation is wrapped with `time.monotonic()` deltas, including the new `metrics` stage from P8 step 1) and emits `dist/build-telemetry.json` containing `{schema_version, $schema, build: {platform, python}, stages: [{stage, duration_ms}], totals: {duration_ms, stages}}`. Stage names are normalised against the `ALL_STAGES` constant; unknown labels fall through unchanged so future stages don't break the schema. **Reproducibility contract — by design the file is _not_ written under `--reproducible`.** Wall-clock timings are inherently non-deterministic and would fail the SOURCE_DATE_EPOCH-style byte-identical promise that other release-time artefacts honour. The audit `tests/build/test_render_telemetry.py::test_render_skips_emission_when_reproducible` pins this; manual verification: two consecutive `tools/build/build.py --reproducible` runs produce zero `dist/build-telemetry.json` files and `dist/metrics.json` byte-identical (sha256 `0812d9a0…`). `schemas/v2/build-telemetry.schema.json` is a JSON Schema 2020-12 contract (`additionalProperties: false`, `duration_ms >= 0`, `stage` non-empty string, `totals.stages` non-negative integer matching `stages.length`). 22 new tests in `tests/build/test_render_telemetry.py` across six dimensions: pure helpers (`_normalise_stage` / `_python_version` / `build_payload`); rendering (top-level + per-stage shape, totals consistency, JSON sort-key stability); reproducibility contract; schema validation (success on canonical payload, rejection on negative durations, unknown top-level fields, missing required keys); pipeline wiring (`render_telemetry` is imported by `build.py`, `metrics` is the last entry in `ALL_STAGES`); cross-references (`SCHEMA_VERSION` matches schema's `version` const; `SCHEMA_REF` matches `$id` suffix; output filename is `build-telemetry.json`, not `telemetry.json`). Total tests: 547 → 569. **Deliberate non-feature:** the schema does not gate per-stage absolute thresholds. Build duration depends on host hardware, IO contention, and Python runtime; absolute thresholds would either flake constantly or be useless. The contract is "every non-reproducible build leaves a parseable record", not "every build must finish in N seconds". Trend analysis (regression detection) is consumer-side and lives outside this PR.
