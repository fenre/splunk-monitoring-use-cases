# ADR-0008: Every constant has exactly one home

- **Status:** Accepted
- **Date:** 2026-05-08
- **Deciders:** Repository maintainers

## Context

Across the v6 → v7 catalog growth, the same data ended up duplicated in
multiple files because each consumer (Python build pipeline, browser
SPA, MCP server, audits) was bolted on independently. As a result we
now have:

| Constant family | Authoritative source today | Duplicates today |
|---|---|---|
| Category metadata (`name`, `description`, `priority`) | `tools/build/enrichment.py:CATEGORIES` | `non-technical-view.js`, `data/categories.json` |
| Equipment / vendor library (105 slugs) | `scripts/equipment_lib.py:EQUIPMENT` | `tools/build/build.py:EQUIPMENT` *(legacy shim)*, `data/equipment.json` |
| Splunkbase<sup class="ref">[<a href="#ref-3">3</a>]</sup> app catalogue (apps + TAs) | `tools/build/enrichment.py:SPLUNKBASE_APPS` | `data/splunkbase.json` *(stale snapshot)* |
| Regulation taxonomy (cat-22 frameworks) | `data/regulations.json` | none — already singleton |
| MITRE ATT&CK<sup class="ref">[<a href="#ref-2">2</a>]</sup> techniques | `data/mitre.json` | none — already singleton |
| UC content | `content/cat-*/UC-*.json` per ADR-0007 | none — already singleton |
| Implementation roadmap (waves) | `tools/build/enrichment.py:IMPLEMENTATION_ROADMAP` | catalog.json (re-emitted), `non-technical-view.js` (curated subset) |

Every duplicate creates one of three drift modes:

1. **Silent semantic drift** — two copies of the same constant disagree;
   downstream artefacts compute different answers depending on which
   copy they read first. Example: a category renamed in the Python
   source but not in `non-technical-view.js` produces a UI that says
   "Network Security" in technical mode and the old name in
   non-technical mode.
2. **Schema drift** — the Python copy gains a field (e.g. `priority`)
   that the JSON copy doesn't represent, so a dependent audit can
   pass against one and fail against the other.
3. **Authoring split-brain** — contributors don't know which copy to
   edit, so PRs touch the wrong one and CI passes anyway because the
   non-authoritative copy is the one consumed.

The codeguard rules require us to treat every duplicate as a
trust-chain weakness: the source-of-truth principle is the only thing
that prevents one of the duplicates from becoming an attacker-supplied
fork that the build silently consumes.

This ADR codifies what is already implicit in the current build
pipeline (and in [ADR-0007](0007-json-as-source-of-truth.md) for UC
content) and extends it to every other constant in the repository.

## Decision

**Every constant has exactly one home.** That home is either:

- a Python module under `tools/build/` or `scripts/`, when the
  constant has rich semantic structure (functions, conditionals,
  cross-references) — example: `enrichment.CATEGORIES`; or
- a JSON file under `data/`, when the constant is pure data with no
  computed dependencies — example: `data/regulations.json`,
  `data/mitre.json`.

**JavaScript twins are always generated from the Python source at
build time.** They are not authored. The build pipeline emits them
into `dist/` (and, transitionally, into the project root for the
files that are still committed there per [ADR-0009](0009-generated-artefact-policy.md))
and the source-of-truth file is the Python module, not the `.js`
output. Specifically:

- `non-technical-view.js` → generated from `tools/build/enrichment.py`'s
  `CATEGORIES` and `IMPLEMENTATION_ROADMAP` plus the per-UC
  `grandmaExplanation` sidecars under `content/`.
- `data.js` → generated from `catalog.json` (until the file is retired
  in P5; tracked by `p5-data-js-retire`).
- `catalog.json` → generated by `tools/build/render_legacy_artifacts.py`
  from the parsed `Catalog` object; the in-memory `Catalog` is in turn
  built from `content/cat-*/UC-*.json` plus the Python constants.

If a generated `.js` file disagrees with its Python source, the `.js`
file is wrong; rerun the generator. CI gates are required (see
[Consequences §Negative](#negative)) to keep the contract honest.

### Routing rules for new constants

When introducing a new constant, the choice between Python and JSON
follows this checklist:

1. **Does the constant have computed structure** (helper functions,
   reverse indices, validation logic, polymorphic accessors)?
   → Python module. JSON cannot host the helper logic and would
   require parallel maintenance in code.
2. **Is the constant authored by humans without engineering review**
   (curators editing a regulation taxonomy, content reviewers tagging
   MITRE techniques)?
   → JSON file. Pull requests stay readable and the schema is a hard
   gate.
3. **Is the constant a snapshot of an external feed** (Splunkbase, NIST
   OSCAL, MITRE ATT&CK)?
   → JSON file under `data/`, ingested by an `ingest/` script. The
   ingest script is the only writer.
4. **Will the constant ever need to be consumed by JavaScript?**
   → That is independent of the routing decision; the answer is "the
   build pipeline emits a JS twin from the Python or JSON source".

A maintainer reviewing a PR that introduces a constant must apply
this checklist explicitly — drift this rule and we are back to the
duplication problems above.

## Consequences

**Positive:**

- Single point of truth eliminates the three drift classes documented
  in §Context. Every audit, every build, every consumer reads the
  same bytes.
- Refactoring a constant (rename, reshape, extend) becomes a single
  edit instead of an expedition through the repo searching for stale
  copies.
- Generated artefacts can be rebuilt deterministically from sources
  (this is the byte-reproducibility property gated by
  `tests/build/test_legacy_artifacts_parity.py`).
- Reduces supply-chain risk: there is exactly one file an attacker
  could target to inject a constant, and that file is in the Python
  trust boundary that already runs through the codeguard rules.
- Makes the typed-models work in [ADR-0007](0007-json-as-source-of-truth.md)
  and the upcoming `p4-consumer-migration` more tractable: every
  consumer of a constant uses the same in-memory shape.

**Negative:**

- Curators who would otherwise hand-edit `non-technical-view.js`
  cannot — they must edit the Python source and rerun
  `make build`. Mitigation: the generator is < 5 seconds; the dev
  flow stays interactive.
- Removing existing duplicates is a measurable migration cost (the
  P1 plan tracks each one as a separate item). Mitigation: we burn
  them down opportunistically, with a CI gate (`audit_repo_consistency.py`)
  that asserts no new duplicates are introduced.
- A bug in the generator silently affects every consumer at once.
  Mitigation: the generator's output is byte-pinned by the parity
  test suite, so any bug also fails CI.

**Required CI gates** (extant or planned):

- `tools/build/test_legacy_artifacts_parity.py` — asserts catalog.json
  and llms*.txt are byte-identical between two consecutive builds
  from the same SSOT inputs.
- `python3 -m splunk_uc audit-repo-consistency` — asserts no new duplicates are
  introduced. Extended in P1 to cover JS twin generation.
- `python3 -m splunk_uc audit-non-technical-sync` — asserts every category in
  `non-technical-view.js` exists in the Python source and vice versa.

## Alternatives considered

- **Embed every constant inline in JavaScript so the SPA can read it
  directly.** Rejected: forces the build pipeline to either parse JS
  (fragile) or duplicate the data in Python; both reintroduce the
  drift modes this ADR exists to eliminate.
- **A single SQLite database** that both Python and JS read via a
  generated WASM build. Rejected: the static-site contract
  ([ADR-0002](0002-static-single-page-app.md)) would break;
  contributors would need infrastructure to view a PR diff.
- **Authoring rules without enforcement** (just write a CONTRIBUTING
  paragraph). Rejected: empirically this fails — the existing
  duplicates were introduced under exactly such an unenforced norm.
