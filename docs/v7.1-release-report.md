## The big idea

v7.0 made every UC its own file; v7.1 makes every UC **explainable
without jargon, everywhere**.

Before this release, flipping the "Non-technical view" toggle changed
the sidebar and the landing page — but the moment a user clicked into
a UC, they saw SPL, CIM data models, MITRE technique IDs, index names,
and tstats queries. Curated plain-language `why` lines in
`non-technical-view.js` covered only ~10 % of the catalogue.

v7.1 closes that gap with a first-class, schema-validated
`grandmaExplanation` on every UC, a deterministic generator that
populates it, a CI guard that blocks merges that forget it, and a
wholesale rewrite of the non-technical UI that uses it *everywhere*
and hides everything technical behind a single disclosure.

## 1. The `grandmaExplanation` field

**What it is:** a new optional string on every UC sidecar, 20–400
characters, one to three sentences, in first-person plural (`we watch…`,
`we catch…`), with no Splunk / SPL / CIM / MITRE / TA / add-on
acronyms and no regulation clause numbers.

**Where it's defined:** `schemas/uc.schema.json` (schema v1.5.0 — see
`schemas/changelogs/uc.md`).

**Why it matters:** it's the first per-UC field designed for
non-technical readers. Every other UC field is curator-oriented
(criticality, wave, MITRE techniques, CIM models…); `grandmaExplanation`
is the first one that reads like a sentence a non-engineer can actually
say out loud.

Example:

```jsonc
{
  "id": "UC-1.1.1",
  "title": "Disk full on Linux server",
  "grandmaExplanation": "We watch every Linux server's disks and warn you before one fills up, so the server doesn't stop working in the middle of the night."
}
```

Full authoring guide: [`docs/grandma-explanations.md`](grandma-explanations.md).

## 2. Non-technical UI rewrite

The non-technical toggle used to swap the sidebar and the category
landing page. In v7.1 it now drives the **entire UI**:

| Surface | Technical mode | Non-technical mode (v7.1) |
|---|---|---|
| **UC card** | Title + technical description + all chips | Title + `ge` sentence + criticality only (technical description / app chips hidden) |
| **Search result** | Full technical chrome | `ge` first, technical chrome hidden |
| **Recently added strip** | Technical description | `ge` sentence |
| **Subcategory list** | Technical description + wave chip | `ge` sentence, clickable row |
| **UC detail panel** | All sections open | `ge` block on top, *every* technical section (SPL, CIM, MITRE, Data sources, App/TA, Required fields, Detailed implementation, etc.) wrapped in **one** `<details>` disclosure labelled *Show technical details* and collapsed by default |
| **Category landing** | Existing technical category view | Curator-authored `non-technical-view.js` "areas" layout with clickable UC rows |

The detail panel's disclosure open/closed state follows the mode
toggle: toggling to non-technical mode collapses every `c-panel-tech`
block on the page without a full re-render, and toggling back expands
them. Technical readers never see a visible affordance — the
disclosure is transparent and its contents render inline.

## 3. Generator + CI guard

`scripts/generate_grandma_explanations.py` is the single writer of
`grandmaExplanation`. Key properties:

- **Deterministic** — byte-identical output on re-runs at the same
  catalogue state.
- **Curator-respecting** — never overwrites a non-empty value unless
  `--force` is passed.
- **Schema-compliant** — enforces the 20–400 char bounds and falls back
  to a short safe sentence if the composed text would violate them.
- **No API keys** — pure-Python stdlib, using a small drop-list of
  jargon patterns plus simple voice conversions.
- **Idempotent** — rerunning without source changes is a no-op.

CI runs `--check` on every PR via `.github/workflows/validate.yml` and
exits non-zero (blocking merge) if any UC is missing or out-of-bounds.
This makes it **structurally impossible** to ship a UC without
plain-language copy going forward.

## 4. Runtime fallback in `build.py`

Not every UC has a sidecar yet — some still live as markdown only.
To keep the UI coherent during the v7.x migration, `build.py` composes a
runtime `ge` for markdown-only UCs from the existing `title` / `value` /
`description` and emits it into `catalog.json` under the same short key
`ge`. The UI cannot tell the two sources apart.

This is **not** a substitute for running the generator — CI's
`--check` still requires every sidecar to have the field. It only
guarantees a non-empty card in the browser for UCs that haven't been
converted to sidecar form.

## 5. Deep links and subcategory navigation

Three small navigation fixes make the non-technical toggle feel
persistent:

1. **`goToSubcat(catId, scId)`** now inspects `nonTechnicalView` and
   dispatches to `renderNonTechnicalSubcategory` when the toggle is on
   — previously clicking a subcategory in the sidebar dropped users
   back into the technical category view.
2. **`renderNonTechnicalSubcategory(catId, scId)`** is the new
   non-technical subcategory renderer. It prefers curator-authored
   `nt-area` blocks (for `whatItIs` / `whoItAffects` / `splunkValue`
   / primer links) and falls back to listing all UCs in the
   subcategory using `uc.ge` for each row.
3. **`restoreFromHash()`** now sets `catShowAllUCs = !nonTechnicalView`
   so that hash deep-links (`#cat-10/10.3`, `#uc/UC-1.1.1`, …) restore
   the right mode when opened cold.

Area-list UC rows are now clickable (open the UC panel via
`openUCById`) and prioritize the fresh `uc.ge` from the catalogue over
stale `why` copy in `non-technical-view.js`.

## What a user sees

- **Flipping the toggle** on any page now changes the UC cards, the
  search results, the recently-added strip, the subcategory list, and
  the UC detail panel all at once — no mixed state.
- **On a UC card** in non-technical mode: a short, warm sentence about
  what the UC watches and why anyone should care. No acronyms, no
  technology names the reader doesn't recognise.
- **In the UC detail panel** in non-technical mode: the plain-language
  sentence at the top, then a greyed *Show technical details* link
  they can expand if they're curious.
- **Deep-linking into a subcategory or UC** with the toggle on stays
  in non-technical mode — browser back/forward and shared URLs behave
  consistently.

## How it all stays in sync

Every time `build.py` runs:

```
1. Build catalog.json, data.js, api/*.json, llms.txt, sitemap.xml   (existing)
2. → For markdown-only UCs, compose a runtime `ge` and emit it        (NEW in v7.1)
```

Every time CI runs (`.github/workflows/validate.yml`):

```
1. All existing audits (structure, IDs, non-technical-view.js sync, …)
2. → python3 scripts/generate_grandma_explanations.py --check         (NEW in v7.1)
3. python3 build.py (fails if data.js / catalog.json would change)
```

This makes drift impossible: a PR that adds a UC without
plain-language copy fails on step 2; a PR that forgets to rebuild
fails on step 3.
