# Implementation ordering: waves and prerequisite UCs

This page documents the two optional per-UC fields that help operators
plan the order in which they roll out use cases, and the curator rubric
for assigning them.

- **`Wave:`** тАФ one of `crawl` / `walk` / `run`. A coarse tier that tells
  an operator whether this UC is foundational (turn it on first), an
  intermediate correlation layer, or an advanced roll-up that depends on
  several crawl/walk signals.
- **`Prerequisite UCs:`** тАФ a comma-separated list of `UC-X.Y.Z` ids
  that must be in place before this UC can work end-to-end. Typical
  reasons: shared lookups, macros, ITSI services, normalised indexes,
  or CIM-compliance work.

Together they power three user-facing surfaces:

1. A **wave badge** on every UC detail panel (`Crawl` / `Walk` / `Run`).
2. An **"Implement first"** list of clickable prerequisite chips inside
   the UC panel, plus a reverse-lookup **"Enables"** list of UCs that
   declare the current UC as a prerequisite.
3. A per-category **Crawl тЖТ Walk тЖТ Run** roadmap above the UC list that
   groups every UC in the category into the three waves (plus an
   `unassigned` bucket for UCs the curator has not tagged yet).

The fields are also exposed through every downstream projection:
`catalog.json` (`implementationRoadmap`, plus compact `wv` / `pre`
keys on each UC), the `/api/v1/` JSON surface, the MCP server's
`search_use_cases` and `get_use_case` tools, and the static per-UC
HTML pages generated by `tools/build/templates/uc.py`.

## 1. Authoring syntax

Add either or both of these lines to the bullet list inside any UC
block in `content/cat-*/UC-*.md` companion prose **or** set the equivalent fields (`wave`, `prerequisiteUseCases`) on `content/cat-*/UC-*.json`. Order does not matter in markdown, but keep them
adjacent to `Criticality` / `Difficulty` so reviewers see them first.

```markdown
### UC-1.1.1 ┬╖ CPU Utilization Trending (Linux)
- **Criticality:** ЁЯЯа High
- **Difficulty:** ЁЯЯв Beginner
- **Wave:** crawl
- **Prerequisite UCs:** N/A
- **Value:** ...
```

```markdown
### UC-1.1.5 ┬╖ CPU Anomaly Detection (Linux)
- **Criticality:** ЁЯФ┤ Critical
- **Difficulty:** ЁЯФ┤ Expert
- **Wave:** run
- **Prerequisite UCs:** UC-1.1.1, UC-1.1.2
- **Value:** ...
```

### Field rules

| Field | Parser behaviour |
|---|---|
| `Wave:` | Case-insensitive. Accepts `crawl`, `walk`, `run`. Emoji prefixes (e.g. `ЁЯЯв crawl`) and the words *Foundation* / *Intermediate* / *Advanced* are accepted as synonyms and normalised to `crawl` / `walk` / `run`. Any other string is stored verbatim and rejected by `python3 -m splunk_uc audit-catalog-schema` with the UC id in the error. |
| `Prerequisite UCs:` | Comma-, whitespace-, or bullet-separated list of `UC-X.Y.Z` tokens. Prose around the tokens is ignored. Tokens that do not match the `UC-\d+\.\d+\.\d+` grammar are silently dropped at parse time **and** rejected by the schema audit, so the build fails twice on typos. Use `N/A` (or omit the line) when the UC has no prerequisites. |

Aliases recognised for the prerequisite field: `Prerequisite UC:`,
`Prerequisites:`. Internally they all resolve to the `pre` compact key
in `catalog.json` and in the built SPA payload.

## 2. Curator rubric

Use this rubric when deciding which wave to assign. When in doubt,
pick the earlier wave тАФ it is always safe for an operator to run a
`crawl` UC first and promote it later.

### 2.1 `crawl` тАФ Foundation

Implement first. A UC belongs in `crawl` when **all** of the following
are true:

- Installing the required TA or turning on the data feed is the bulk
  of the work.
- A single panel, saved search, or alert delivers immediate value.
- It depends on **no other UC** in the repository (empty
  `Prerequisite UCs:`).
- A first-time Splunk admin can stand it up in an afternoon.

Typical `crawl` UCs: "CPU Utilization Trending", "Disk Space Monitoring",
"Authentication Failures by Source", "Firewall Drop Rate".

### 2.2 `walk` тАФ Intermediate

A UC belongs in `walk` when **any** of the following hold:

- It **refines** a crawl signal (anomaly thresholds, seasonality,
  z-scores, EWMA, percentiles).
- It **correlates** two or more crawl signals into a single actionable
  event (e.g. high CPU *and* high disk I/O on the same host).
- It introduces a lookup, macro, or summary index that a handful of
  downstream `run` UCs will consume.
- It maps its output into CIM or a data model so that later UCs can
  run `tstats`.

Typical `walk` UCs: "CPU Anomaly Detection by Host", "SLA Math over
Login Success", "Capacity Forecast (7-day regression)".

### 2.3 `run` тАФ Advanced

A UC belongs in `run` when **any** of the following hold:

- It depends on **multiple** crawl or walk UCs (cross-category is
  common: e.g. endpoint + network + identity).
- It produces an ITSI service, a risk-based alert, or a composite KPI
  that downstream dashboards consume.
- Rolling it out to a noisy production environment requires a
  multi-sprint tuning plan.
- It cannot be understood without the broader security / operations
  narrative of its category.

Typical `run` UCs: "Impossible Travel with Device Posture", "MITRE
ATT&CK<sup class="ref">[<a href="#ref-3">3</a>]</sup> T1078 Correlation Across EDR / IdP / CASB", "Service Health
Score for Core Banking".

### 2.4 Unassigned

UCs without a `Wave:` line appear in the category roadmap's
`unassigned` counter. This is a curator TODO signal, not an error тАФ
the catalogue ships with most UCs unassigned and we seed waves
incrementally.

## 3. Prerequisite rubric

A prerequisite is a **hard technical dependency**. Ask:

> If I turned off UC-X today, would UC-Y still function?

If the answer is "no тАФ the lookup goes stale, the macro breaks, the
data model stops being accelerated, etc.", then UC-X is a prerequisite
of UC-Y.

### 3.1 When to add a prerequisite

- UC-Y's SPL references a **lookup** or **macro** authored by UC-X.
- UC-Y relies on CIM fields that UC-X populates via `eval` / `rex` /
  field extractions (rare in this repository because CIM fields are
  TA-level, but it happens with custom schemas).
- UC-Y runs against an **ITSI service** or **summary index** that
  UC-X defines.
- UC-Y's detection logic assumes a **baseline** computed by UC-X (for
  example, a weekly baseline search that UC-Y compares the live
  stream to).

### 3.2 When **not** to add a prerequisite

- UC-Y and UC-X share a **data source** but read it independently.
  (A shared TA is not a prerequisite тАФ it is a deployment concern that
  belongs in `App/TA:`.)
- UC-X is merely "related" or "good to know about". Use the
  narrative `Implementation:` block to cross-link, or add both UCs to
  the same `Regulations:` cluster тАФ but do **not** add them to `pre`.
- UC-X is in a sibling category but you *would like* operators to
  see them together. That is what the category-level roadmap and the
  filter facets are for.

### 3.3 Cycle rule

The prerequisite graph must be a DAG.
`python -m splunk_uc audit-prerequisites` runs iterative DFS cycle detection on
every PR and will reject a cycle with a concrete path in the error
message. Refactor by introducing a shared intermediate UC, or by
demoting one of the two to a `run` UC that consumes the other.

### 3.4 Wave monotonicity

An `earlier` wave should never depend on a `later` wave тАФ a `crawl`
UC that declares a `run` UC as a prerequisite is almost certainly a
curator error (the waves were inverted). The audit raises this as a
warning by default; `--strict` mode promotes it to a hard failure, and
releases enable `--strict` via the release workflow.

## 4. Validation and CI gates

All three gates run on every pull request:

| Gate | Script | Behaviour |
|---|---|---|
| Build-time parser + warnings | `tools/build/build.py` (prerequisite validation) | Rejects unknown ids, self-references, and cycles; prints wave distribution and monotonicity warnings. |
| Schema validation | `python -m splunk_uc audit-catalog-schema` | Enforces `wv тИИ {crawl, walk, run}`, the `UC-X.Y.Z` grammar on every `pre[]` entry, no duplicates, and no self-references. |
| Graph audit | `python -m splunk_uc audit-prerequisites --check` | Re-runs every graph invariant against the committed `catalog.json`, diffs the deterministic report at `reports/prerequisites-audit.json`, and fails on drift. |

Local invocation:

```bash
make build                                                 # emits fresh catalog.json + dist/ + reports/
PYTHONPATH=src python3 -m splunk_uc audit-catalog-schema                 # per-UC shape checks
PYTHONPATH=src python3 -m splunk_uc audit-prerequisites --write-report   # regenerate the graph audit report
PYTHONPATH=src python3 -m splunk_uc audit-prerequisites --check          # CI-parity drift check
```

## 5. Downstream surfaces

Once the markdown is updated and the build has run, the waves and
prerequisites appear automatically in:

- `index.html` тАФ wave badge on the UC panel; "Implement first" +
  "Enables" chip lists; per-category `Crawl тЖТ Walk тЖТ Run` roadmap band.
- `catalog.json` тАФ compact `wv` and `pre` per UC, plus the top-level
  `implementationRoadmap` object keyed by category.
- Built SPA (under `dist/`) тАФ same payload projected into globals
  (`DATA`, `ROADMAP`) for the catalog UI.
- `api/v1/uc-thin.json` and `api/v1/compliance/*.json` тАФ the compact
  wave / prerequisite arrays for downstream agents.
- `tools/build/templates/uc.py` тАФ the static per-UC page renders the
  wave badge, prerequisite chips, and reverse-lookup "Enables" list.
- MCP server тАФ `search_use_cases` accepts a `wave` filter and the
  response includes `wave` and `prerequisiteUseCases`; `get_use_case`
  exposes both fields plus the reverse `enables` list.
