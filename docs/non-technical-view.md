# Non-technical View — Reader Guide

Audience: executives, business stakeholders, privacy officers, legal,
auditors, board members, and anyone who needs to understand *what* the
catalogue covers without reading SPL or wading through CIM models.

This guide explains what the **Non-technical / Technical** toggle in the
header changes and where the plain-language copy comes from. For the
authoring contract that produces this content see
[Plain-language Explanations](grandma-explanations.md).

## What the toggle does

The header carries a two-state switch:

- **Technical** (default) — the full catalogue with every filter, SPL,
  CIM model, regulation chip, and audit panel. Built for SOC analysts,
  IT operators, and detection engineers. See the
  [Site User Guide](site-user-guide.md) for the technical UX.
- **Non-technical** — a re-skinned home page with plain-language outcomes,
  area summaries, and curated example use cases. Built for the readers
  named above.

The choice persists in `localStorage` and survives reloads. Each view
has its own URL fragment scheme so you can share a link that opens
directly in the non-technical mode.

## What you see in non-technical mode

### Outcomes panel

Each of the 23 categories opens with a **2–4 sentence outcomes block** —
plain English, no jargon. Examples:

> "We watch your servers and tell you when one is sick before users
> notice."

> "We help you prove to auditors that the right people accessed the
> right things at the right time, and nobody else did."

These come from `non-technical-view.js` (`window.NON_TECHNICAL[<n>].outcomes`)
and are written by humans, not generated.

### Areas of focus

Each subcategory becomes an **area** card with:

- A short, plain-language **name**.
- A 1–2 sentence **description** of what the area watches.
- Up to three **representative example use cases** ("Why this matters
  in plain language").

For category 22 (regulatory compliance), areas additionally carry:

- **What it is** — one-sentence definition of the regulation/control family.
- **Who it affects** — plain-language scope (size, sector, jurisdiction).
- **What we deliver** — what the catalogue specifically provides for this regulation.
- A link to the [Regulatory Primer](regulatory-primer.md) for the long story.
- A link to the [Evidence Pack](evidence-packs/) for the auditor-facing rollup.

### Per-UC plain language

Every individual use case in the catalogue carries a
**`grandmaExplanation`** field. In non-technical mode this short
plain-language sentence is rendered:

- At the top of the **detail panel** (as the headline "In plain language" block).
- On every **UC card** as the description.
- In **search results**.
- In the **recently added** list.
- In the **subcategory area** view.

If a UC has both a curated **`why`** statement in `non-technical-view.js`
and an authored **`grandmaExplanation`**, the live `grandmaExplanation`
wins so non-technical readers always see the freshest sentence.

### Outcomes search

The non-technical home has its own filter input that searches across
outcomes, area names, and area descriptions. Use it the same way as the
technical search.

### Hidden in non-technical mode

To keep the surface uncluttered, non-technical mode hides:

- All technical filter chips (CIM, MITRE, ESCU, app/TA, detection type, premium apps).
- The MITRE Coverage Map button.
- The advanced filter strip.
- The Quality and Recently Added tabs (the latter is folded into the area cards).
- The KPI strip's developer-oriented numbers (it shows count of areas instead).
- The Sort dropdown and CSV/JSON exports.

The detail panel still shows everything, because business stakeholders
sometimes need to drill into the technical content. Use the audience
toggle in the header to switch back.

## Where the copy lives

| Surface | Source |
|---|---|
| Category outcomes | `non-technical-view.js` → `outcomes[]` (manual) |
| Area name / description / regulatory fields | `non-technical-view.js` → `areas[]` (manual) |
| Curated "why" statements | `non-technical-view.js` → `areas[].ucs[].why` (manual, optional) |
| Per-UC plain-language sentence (`ge`) | `content/cat-NN-*/UC-X.Y.Z.json` → `grandmaExplanation` (generated by `python -m splunk_uc generate-grandma-explanations`, then human-reviewable) |

The **single rule** governing the relationship between the curated copy
and the per-UC field is in the workspace rule
[`non-technical-sync.mdc`](../.cursor/rules/non-technical-sync.mdc). The
authoring contract for `grandmaExplanation` is in
[Plain-language Explanations](grandma-explanations.md).

## Quality bar for non-technical copy

These rules are enforced in CI and review:

- No Splunk / SPL / CIM / MITRE / ITSI / ES / SOAR / regulatory
  acronyms without an explanation.
- Use the "we" voice: "We watch…", "We help you…", "We detect…".
- Outcomes: 2–4 sentences. Area descriptions: 1–2 sentences.
  `grandmaExplanation`: 20–400 characters, one short sentence.
- Each area lists 3 representative UCs (2 acceptable for very small
  subcategories).

## Validation

After editing `non-technical-view.js` or `grandmaExplanation` fields:

```bash
node -e "const window = {}; eval(require('fs').readFileSync('non-technical-view.js','utf8')); console.log(Object.keys(window.NON_TECHNICAL).length + ' categories OK');"
PYTHONPATH=src python3 -m splunk_uc generate-grandma-explanations --check
make build
```

The `--check` mode for the generator blocks PRs where any
`grandmaExplanation` field has drifted from the rule (length, tone, or
banned acronym) — see the [P6 dispatcher reference](scripts-taxonomy.md).
