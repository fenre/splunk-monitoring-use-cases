# `api/` — generated HTTP/JSON surface

> **Heads-up:** Everything under this directory **other than this
> README** is **machine-generated** by `scripts/generate_api_surface.py`
> on every build. The whole tree is in `.gitignore` (see the rule at
> the bottom) — only this README is checked in (via a `!api/README.md`
> exception). Do not edit any other file under `api/` by hand; your
> change will disappear the next time `python3 build.py` runs.

## Where the data comes from

```
use-cases/cat-*/uc-*.json    ┐
data/regulations.json         │      scripts/generate_api_surface.py
data/equipment-models.json    │ ───────────────────────────────────►   api/v1/**
data/mitre/**                 │
... (read-only inputs) ─────┘
```

The generator is invoked automatically by:

- `python3 build.py` — local rebuild (idempotent; rerun twice should be
  byte-identical).
- `.github/workflows/validate.yml` — CI runs
  `python3 scripts/generate_api_surface.py --check` and fails the PR if
  the regenerated tree drifts from a fresh clone.

## What lives here at runtime

```
api/
├── index.json              # category-level rollup (regenerated by build.py)
├── cat-<n>.json            # one per use-case category (1..23)
└── v1/                     # versioned compliance + equipment + MITRE + OSCAL surface
    ├── README.md           # versioned quick start (regenerated by generate_api_surface.py)
    ├── manifest.json       # endpoint catalogue
    ├── openapi.yaml        # OpenAPI 3.1 spec
    ├── context.jsonld      # JSON-LD context for the schema
    ├── compliance/         # coverage, gaps, regulations, UCs
    ├── equipment/          # per-equipment UC + regulation maps
    ├── evidence-packs/     # per-regulation evidence bundles
    ├── mitre/              # MITRE ATT&CK techniques + tactics index
    ├── oscal/              # OSCAL component-definitions per UC
    └── recommender/        # UC recommender corpus + index
```

For the full endpoint catalogue and `curl` examples, see the generated
`api/v1/README.md` after running `python3 build.py`, or the live deployment
at <https://fenre.github.io/splunk-monitoring-use-cases/api/v1/>.

## Why is the data gitignored?

- The data is **derivable** from the in-repo source of truth
  (`use-cases/`, `data/`, `schemas/`). Carrying it in git would double
  the size of the repo and create constant noise on every PR.
- CI guarantees the deployed copy matches the inputs by re-running the
  generator with `--check`.
- GitHub Pages serves the regenerated tree on every push to `main`, so
  consumers always get the version that matches the latest sources.

## How to consume the API

```bash
# From a clone (after `python3 build.py`):
jq '.endpoints | keys' api/v1/manifest.json

# From the live deployment:
curl -s https://fenre.github.io/splunk-monitoring-use-cases/api/v1/manifest.json | jq '.endpoints | keys'
```

See [`docs/api-versioning.md`](../docs/api-versioning.md) for the
backwards-compatibility contract.
