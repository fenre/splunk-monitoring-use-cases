# Sample events for use-case testing

This tree contains representative raw log events (fixtures) used by
`scripts/run_uc_tests.py` (see `v6.0-14`) to verify that shipped SPL actually
runs and returns sensible results when fed known inputs.

## Directory layout

```
samples/
├── README.md                        # this file
├── _schema/
│   └── sample-manifest.schema.json  # JSON Schema for manifest.yaml
└── UC-<category>.<sub>.<uc>/        # one directory per UC
    ├── manifest.yaml                # metadata for the test runner
    ├── positive.log                 # events that SHOULD be returned by the SPL
    └── negative.log                 # (optional) events that must NOT be returned
```

The UC directory name matches the `i` field in `catalog.json`, prefixed with
`UC-`. For example, the CPU-utilization UC `1.1.1` lives under
`samples/UC-1.1.1/`.

## manifest.yaml fields

Each `manifest.yaml` captures what the test runner needs to know:

```yaml
uc_id: "1.1.1"                  # required — matches catalog.json DATA[].s[].u[].i
sourcetype: "cpu"               # required — Splunk sourcetype of positive.log
index: "os"                     # required — index the events will be indexed into
source: "cpu.sh"                # optional — value for Splunk source field
host: "linux-prod-01"           # optional — value for Splunk host field
timerange: "-7d@d"              # optional — search window to use (default: -24h)
expected:
  min_count: 1                  # required — min events that must match the SPL
  max_count: 10                 # optional — upper bound on matching events
  fields:                       # optional — field values expected in results
    - name: host
      values: ["linux-prod-01"]
    - name: avg_cpu
      min: 90
origin: "hand-authored"         # hand-authored | vendor-doc | synthetic | contributor
origin_ref: "https://docs.splunk.com/..."   # optional citation URL
last_reviewed: "2026-04-16"
reviewer: "@splunk-monitoring-use-cases"
```

See `_schema/sample-manifest.schema.json` for the machine-readable contract
and authoritative list of required / optional fields.

## How events are timestamped

`scripts/run_uc_tests.py` rewrites timestamps in `positive.log` to be within
the last 5 minutes before ingesting them, so your sample can use any
ISO-8601 / RFC3164 timestamp as long as `props.conf` for the given sourcetype
can parse it. The test runner will:

1. Read every line as one event.
2. Replace the line's timestamp (per the sourcetype's `TIME_FORMAT`) with
   `now - jitter`.
3. HEC-post the rewritten event into the target index/sourcetype.
4. Run the UC's SPL with the `timerange` from the manifest.
5. Assert on `expected.min_count`, `expected.max_count`, and `expected.fields`.

## Authoring guidelines

**Source your events**, never fabricate them. Acceptable origins:

- **`vendor-doc`** — copied verbatim (or with trivial field substitution for
  hostnames / IPs) from the vendor's official documentation, knowledge base
  or sample-data repo. Always link in `origin_ref`.
- **`hand-authored`** — constructed from first principles by an engineer who
  can verify the fields match what the actual sourcetype produces. Use
  sparingly and document reasoning in `origin_ref` (link to the sourcetype's
  schema doc).
- **`synthetic`** — generated by a trusted tool (e.g. Splunk's `eventgen`,
  vendor-provided `generate_samples.py`). Document the generator in
  `origin_ref`.
- **`contributor`** — sanitized real events donated by a community member.
  Must be scrubbed of IPs, usernames, hostnames, URLs and any PII before
  commit.

**Never** paste real production data without scrubbing. PII, customer data,
internal hostnames, API keys, auth tokens, and private IP ranges must all be
replaced with clearly synthetic placeholders (e.g. `10.0.0.0/8`, `example.com`,
`user-42`).

## Coverage status

Each UC falls into one of three coverage tiers:

- **Tier 1 — golden fixture**: hand-authored by the maintainers, validated in
  CI on every push. This is the gold standard.
- **Tier 2 — contributor fixture**: authored by the community, passing CI,
  reviewed by a maintainer.
- **Tier 3 — stub**: directory exists with a `manifest.yaml` noting the
  expected sourcetype but no `positive.log` yet. These are tracked in the
  coverage report for future contributors.

`scripts/samples_index.py` generates `docs/samples-coverage.md` showing which
UCs are in which tier.

## Contributing a new sample

1. Pick an uncovered UC from `docs/samples-coverage.md`.
2. Create `samples/UC-<id>/manifest.yaml` using the schema below.
3. Add `positive.log` with 1–5 events that should match the SPL.
4. (Optional) Add `negative.log` with 1–5 events that must *not* match.
5. Run `python3 scripts/samples_index.py --validate` locally.
6. Open a pull request.

## Relationship to other files

- `catalog.json` is the source of truth for the set of UCs; sample directory
  names must correspond to a UC listed there.
- `provenance.json` (from `v6.0-16`) aggregates the `origin` + `origin_ref`
  fields so the dashboard can show a "sample source" tooltip per UC.
- The v6.0 scorecard (`v6.0-17`) uses coverage tiers as one of its inputs.
